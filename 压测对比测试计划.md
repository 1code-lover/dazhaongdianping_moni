# hm-dianping 压测对比测试计划（先基线，后优化）

## 1. 测试目标

- 先得到优化前基线数据，再做二级缓存改造，最后做同条件复测。
- 用可量化指标证明优化有效，避免“体感变快”但无法论证。
- 形成一份可用于面试展示的工程化证据链（方案 -> 数据 -> 结论）。

## 2. 本轮要验证的核心假设

1. 引入 L1 本地缓存后，`/shop/{id}`、`/shop-type/list` 的 TP95 延迟下降。
2. 引入 L1 本地缓存后，Redis QPS 与 Redis CPU 使用率下降。
3. 在更新操作存在时，缓存一致性可接受（最终一致，秒级收敛）。

## 3. 测试范围与场景矩阵

| 场景ID | 接口 | 目的 | 负载类型 | 鉴权 |
|---|---|---|---|---|
| S1 | `GET /shop/{id}` | 验证热点店铺查询读性能 | 纯读热点 | 否 |
| S2 | `GET /shop-type/list` | 验证低变更列表缓存收益 | 纯读热点 | 否 |
| S3 | `GET /shop/of/type?typeId=1&current=1` | 验证分页列表在缓存参与下表现 | 读多写少 | 否 |
| S4 | `PUT /shop` + `GET /shop/{id}` | 验证更新后读一致性与抖动 | 读写混合 | 是（按项目实际） |
| S5 | `POST /voucher-order/seckill/{id}` | 保留业务链路对照（非本次优化重点） | 高并发写 | 是 |

> 说明：本次二级缓存优化重点看 S1/S2，S3/S4 用来补充“真实业务可用性”，S5 作为旁证观察是否被副作用影响。

## 5. 数据准备与预热

## 5.1 数据准备

- 准备热点 shop ID（如前 20 个高频店铺），用于 S1。
- 准备 1~2 个常用 typeId，用于 S3。
- 若 S4/S5 需登录，提前生成 token 池（可复用你仓库已有 token 文件方案）。

## 5.2 预热策略

- 每轮正式压测前，先做 3~5 分钟预热流量。
- 对“冷缓存测试”与“热缓存测试”分开执行并单独记录：
  - 冷缓存：清理 Redis/L1 后开测
  - 热缓存：预热后开测

## 6. 负载模型（统一模板）

每个场景都按以下节奏执行：

1. **Ramp-up**：2 分钟从 0 拉升到目标并发。
2. **Steady**：稳定压测 10 分钟。
3. **Ramp-down**：1 分钟降载。
4. **重复**：同场景跑 3 轮。

并发梯度建议（按机器能力可下调）：

| 轮次 | 并发用户数 | 目标 |
|---|---|---|
| L1 | 50 | 稳态基线 |
| L2 | 100 | 常规压力 |
| L3 | 200 | 接近瓶颈 |
| L4 | 300 | 观察退化曲线 |

> 若本机资源不足，可改为 30/60/120/180，但前后必须一致。

## 7. 指标口径（前后必须同口径）

## 7.1 应用侧

- QPS / TPS
- 平均响应时间
- TP90 / TP95 / TP99
- 错误率（HTTP 非 2xx + 业务失败率）
- JVM：Heap 使用率、Young/Full GC 次数与耗时、线程数

## 7.2 Redis 侧

- `instantaneous_ops_per_sec`
- CPU 使用率
- 内存占用与 key 数量变化
- 命中率（`keyspace_hits`, `keyspace_misses`）

## 7.3 MySQL 侧

- QPS
- 慢查询数量
- 平均查询耗时

## 8. 执行步骤（严格按顺序）

1. 记录环境信息表（第 4 节）。
2. 执行优化前压测（Baseline）：
   - 依次跑 S1~S5
   - 每场景按 L1~L4，且每个梯度 3 轮
3. 导出原始结果（JMeter `.jtl` + 汇总 CSV）。
4. 实施二级缓存改造。
5. 在同环境执行优化后压测（After）。
6. 生成对比报告并给出结论（收益与副作用都要写）。

## 10. 达标标准（建议）

- S1/S2：
  - TP95 至少下降 20%
  - 错误率不高于 Baseline（目标 <= 0.5%）
  - Redis OPS 至少下降 15%
- S4：
  - 更新后读取一致性在可接受时间窗内（如 <= 2 秒）
- S5：
  - 不出现明显回归（成功率与延迟不显著劣化）

## 11. 典型风险与规避

- **风险1：前后环境不一致**  
  规避：锁定机器与配置，所有参数写入第 4 节。

- **风险2：只看平均值，忽略长尾**  
  规避：主看 TP95/TP99，不只看 Avg。

- **风险3：压测脚本差异导致不可比**  
  规避：同一 `.jmx`，仅修改必要变量（如目标地址/并发）。

- **风险4：缓存预热状态不一致**  
  规避：冷/热缓存分开测，并分别记录。

## 12. JMeter 执行建议（结合你现有文件）

- 优先复用仓库里的 `秒杀抢购.jmx`。
- 在同一个 Test Plan 中为 S1~S5 分 Thread Group（或复制组），每组单独控制并发梯度。
- 命令行执行示例（按你本机路径调整）：

```bash
jmeter -n -t 秒杀抢购.jmx -l baseline_s1_l2.jtl -e -o report_baseline_s1_l2
```

> Windows 下建议固定同一版本 JMeter，并关闭不必要后台程序，减少噪声。

## 13. 面试输出建议（1 分钟版本）

- 我先做了优化前基线压测，明确瓶颈在热点读接口和 Redis 压力。
- 然后引入二级缓存（L1 Caffeine + L2 Redis），并通过统一压测模型做同条件复测。
- 数据上重点看 TP95、错误率和 Redis OPS，结果显示热点查询延迟下降、Redis 压力下降且业务错误率无上升。
- 同时补了读写一致性验证，证明方案不是“只快不稳”。
