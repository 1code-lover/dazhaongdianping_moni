## 2. 点赞ZSet详解

### 2.1 ZSet数据结构介绍

Redis ZSet（有序集合）是**Sorted Set**的数据结构，它有三个核心特性：

1. **去重**：每个成员（member）只能出现一次
2. **有序**：成员按照score（分数）进行排序
3. **高效**：插入、删除、查找都是O(logN)复杂度

ZSet的底层实现是**跳表（Skip List）** + **哈希表（Hash Table）**的组合，这使得它既能快速定位成员，又能保持有序。

### 2.2 项目中点赞功能实现

在 `BlogServiceImpl.java:98-122` 中实现了点赞功能：

```java
@Override
public Result likeBlog(Long id) {
    // 1.获取当前登录用户ID
    Long userId = UserHolder.getUser().getId();
    
    // 2.构造Redis key
    String key = BLOG_LIKED_KEY + id;  // "blog:liked:" + blogId
    
    // 3.检查用户是否已点赞（用score判断）
    Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());
    
    if (score == null) {
        // ===== 未点赞，执行点赞操作 =====
        
        // 3.1.数据库点赞数+1
        boolean isSuccess = update().setSql("liked = liked + 1").eq("id", id).update();
        
        // 3.2.将用户ID添加到ZSet
        // 参数：key, member, score（这里用时间戳作为分数，可用于排序）
        if (isSuccess) {
            stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
        }
    } else {
        // ===== 已点赞，取消点赞 =====
        
        // 4.1.数据库点赞数-1
        boolean isSuccess = update().setSql("liked = liked - 1").eq("id", id).update();
        
        // 4.2.从ZSet中移除用户
        if (isSuccess) {
            stringRedisTemplate.opsForZSet().remove(key, userId.toString());
        }
    }
    return Result.ok();
}
```

### 2.3 为什么选用ZSet而不选用Set

| 特性 | Set | ZSet | 选择ZSet的原因 |
|-----|-----|------|----------------|
| 去重 | ✓ | ✓ | 防止重复点赞 |
| 排序 | ✗ | ✓ | 可展示"最近点赞的用户" |
| 判断是否点赞 | O(N) | O(1) | 高效判断 |
| 获取点赞列表 | O(N) | O(logN+N) | 快速获取Top用户 |
| 内存占用 | 较低 | 较高 | 值得 |
| 写入性能 | O(1) | O(logN) | 可接受 |

### 2.4 ZSet的典型使用场景

```
ZSet的典型使用场景：

1. 排行榜
   - 积分榜：用score存储积分，按积分排序
   - 热搜榜：用score存储热度，按热度排序
   
2. 权重队列
   - 优先级任务队列：score表示优先级
   
3. 去重+排序
   - 点赞列表：用户ID去重，按时间排序
   - 访问记录：用户ID去重，按时间排序
```

### 2.5 ZSet命令对照表

| 命令 | 说明 | 时间复杂度 |
|-----|------|-----------|
| ZADD key score member | 添加成员 | O(logN) |
| ZREM key member | 删除成员 | O(logN) |
| ZSCORE key member | 获取成员分数 | O(1) |
| ZRANK key member | 获取排名（从小到大） | O(logN) |
| ZREVRANK key member | 获取排名（从大到小） | O(logN) |
| ZRANGE key start stop | 获取排名范围 | O(logN+N) |
| ZRANGEBYSCORE key min max | 按分数范围获取 | O(logN+N) |
| ZCARD key | 获取成员数量 | O(1) |
| ZCOUNT key min max | 统计分数范围内的成员 | O(logN) |

## 3. 排行榜大-key改进方案

### 3.1 问题分析

当某个热门博客有100万点赞时，`blog:liked:{id}` 这个key会非常大，导致以下问题：

#### 3.1.1 内存问题

```
单个ZSet最大可以存储约2^32个成员（约40亿）
但实际场景中：
- 100万成员 ≈ 150MB内存
- 1000万成员 ≈ 1.5GB内存
- 1亿成员 ≈ 15GB内存
```

#### 3.1.2 性能问题

```
Redis是单线程模型，大key操作会阻塞主线程：
- ZADD: O(logN) → N很大时变慢
- ZRANGE: O(logN+N) → 返回大量数据
- ZCARD: O(1) → 但DEL大key会阻塞
```

#### 3.1.3 热度度不均问题

```
20%的博客占据80%的点赞量：
- 头部博客：100万点赞
- 腰部博客：1万点赞
- 尾部博客：100点赞
```

#### 3.1.4 网络问题

```
大key传输会占用大量网络带宽：
- 读取100万成员 ≈ 100MB网络传输
- 可能导致TCP缓冲区溢出
```

### 3.2 改进方案一：Key拆分（分片）

#### 3.2.1 按时间分片

**思路**：按月份或年份拆分key，定期归档历史数据

```
原有key（压力大）：
  blog:liked:1001 → [user1, user2, ..., user1000000]

改进后（分散压力）：
  blog:liked:1001:2024-01 → 1月点赞用户
  blog:liked:1001:2024-02 → 2月点赞用户
  blog:liked:1001:2024-03 → 3月点赞用户
  ...
  blog:liked:1001:2024-12 → 12月点赞用户
```

**代码实现**：

```java
// 月份分片key生成
public String getLikeKey(Long blogId, LocalDateTime dateTime) {
    String month = dateTime.format(DateTimeFormatter.ofPattern("yyyy-MM"));
    return "blog:liked:" + blogId + ":" + month;
}

// 点赞时使用当前月份key
public void likeBlog(Long blogId, Long userId) {
    String key = getLikeKey(blogId, LocalDateTime.now());
    stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
}

// 获取总点赞数（合并所有月份）
public Long getTotalLikes(Long blogId) {
    String pattern = "blog:liked:" + blogId + ":*";
    Set<String> keys = stringRedisTemplate.keys(pattern);
    long total = 0;
    for (String key : keys) {
        total += stringRedisTemplate.opsForZSet().zCard(key);
    }
    return total;
}
```

**优点**：
- 单个key数据量大幅降低
- 支持历史数据归档（删除旧月份key）
- 减少内存压力

**缺点**：
- 查询总数需要遍历所有分片
- 代码复杂度增加

#### 3.2.2 按用户ID范围分片

**思路**：将用户ID取模，分配到不同的key

```
假设分10片（取模）：
  blog:liked:1001:0  → userId % 10 == 0 的用户
  blog:liked:1001:1  → userId % 10 == 1 的用户
  ...
  blog:liked:1001:9  → userId % 10 == 9 的用户
```

**代码实现**：

```java
// 分片数量
private static final int SHARD_COUNT = 10;

// 获取分片key
public String getShardKey(Long blogId, Long userId) {
    int shard = (int) (userId % SHARD_COUNT);
    return "blog:liked:" + blogId + ":" + shard;
}

// 点赞
public void likeBlog(Long blogId, Long userId) {
    String key = getShardKey(blogId, userId);
    stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
}

// 检查是否点赞（需要检查所有分片）
public boolean isLiked(Long blogId, Long userId) {
    for (int i = 0; i < SHARD_COUNT; i++) {
        String key = "blog:liked:" + blogId + ":" + i;
        Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());
        if (score != null) {
            return true;
        }
    }
    return false;
}
```

**优点**：
- 数据均匀分布
- 单key大小可控

**缺点**：
- 查询是否点赞需要遍历所有分片

#### 3.2.3 复合分片（推荐）

**思路**：结合时间分片 + ID分片，解决热点和历史问题

```
热数据（最近6个月，按ID分片10片）：
  blog:liked:1001:hot:0  → userId % 10 == 0
  blog:liked:1001:hot:1  → userId % 10 == 1
  ...

冷数据（超过6个月归档）：
  blog:liked:1001:archive:2024-01
  blog:liked:1001:archive:2024-02
  ...
```

### 3.3 改进方案二：存储结构优化

#### 3.3.1 用Hash替代ZSet

如果不需要"谁点赞了"的列表，只关心"点赞数"和"是否点赞"：

```
blog:liked:1001 → {
    user1: 1,
    user2: 1,
    user3: 1
}
```

**优点**：
- O(1)判断是否点赞
- O(1)点赞/取消点赞
- 内存占用更低（无需score）

**缺点**：
- 无法获取点赞列表
- 无法按时间排序

#### 3.3.2 用Bitmap

如果用户ID是整数且连续，可以用Bitmap：

```
blog:liked:1001 → 1111010101... (bit数组)
```

**优点**：
- 极度节省内存（1亿用户仅12.5MB）
- O(1)判断和设置

**缺点**：
- 需要连续的用户ID空间

### 3.4 改进方案三：分层存储

#### 3.4.1 冷热分离

```
热数据（7天内）：
  Redis ZSet → blog:liked:1001:recent

冷数据（历史）：
  MySQL → tb_blog_liked_log
  
定时归档：
  每天凌晨将7天前的数据从Redis迁移到MySQL
```

#### 3.4.2 数据库冗余

```
点赞记录：
  1. Redis ZSet → 实时查询
  2. MySQL → 持久化存储
  
定时同步：
  每分钟从Redis同步到MySQL
```

### 3.5 多节点部署方案

项目是无状态服务，可以水平扩展到多节点。本地缓存会出现不一致问题：

#### 3.5.1 问题描述

```
部署3个节点：
  节点A：本地缓存 {blog:1001}
  节点B：本地缓存 {blog:1001}  
  节点C：本地缓存 {blog:1001}

用户点赞 → 节点A修改数据库和Redis → 节点B/C缓存未更新
```

#### 3.5.2 解决方案对比

| 方案 | 原理 | 优点 | 缺点 |
|-----|------|------|------|
| **分布式缓存** | 统一用Redis，本地不做缓存 | 简单可靠 | 性能稍低 |
| **缓存广播** | 缓存更新时Pub/Sub广播 | 实时一致 | 复杂 |
| **一致性哈希** | 同一数据走同一节点 | 无需广播 | 破坏无状态 |
| **本地缓存+TTL** | 短TTL，定期过期 | 简单 | 有延迟 |

**项目当前方案**：使用 **Redis Pub/Sub 广播机制**（见第7节）

### 3.6 完整改进方案总结

```
高点赞量博客的完整方案（推荐）：

1. 分片存储
   blog:liked:{id}:{shard} (10个分片)

2. 冷热分离
   - 热数据：Redis ZSet（7天内）
   - 冷数据：MySQL（历史）

3. 计数优化
   - 实时计数：Redis Hash
   - 定期落库：每分钟同��

4. 缓存策略
   - 本地：Caffeine（热点博客）
   - Redis：通用缓存
   - DB：持久化
```

```java
@Override
public Result follow(Long followUserId, Boolean isFollow) {
    // 1.获取当前登录用户（关注者）
    Long userId = UserHolder.getUser().getId();
    String key = "follows:" + userId;
    
    // 2.判断是关注还是取关
    if (isFollow) {
        // ===== 关注操作 =====
        
        // 2.1.创建关注记录
        Follow follow = new Follow();
        follow.setUserId(userId);
        follow.setFollowUserId(followUserId);
        boolean isSuccess = save(follow);
        
        // 2.2.添加到Redis Set（用于快速查询）
        if (isSuccess) {
            stringRedisTemplate.opsForSet().add(key, followUserId.toString());
        }
    } else {
        // ===== 取关操作 =====
        
        // 3.1.删除数据库记录
        boolean isSuccess = remove(new QueryWrapper<Follow>()
                .eq("user_id", userId)
                .eq("follow_user_id", followUserId));
        
        // 3.2.从Redis Set移除
        if (isSuccess) {
            stringRedisTemplate.opsForSet().remove(key, followUserId.toString());
        }
    }
    return Result.ok();
}
```

```java
@Override
public Result followCommons(Long targetUserId) {
    // 1.获取当前用户Key
    String currentKey = "follows:" + UserHolder.getUser().getId();
    
    // 2.获取目标用户Key
    String targetKey = "follows:" + targetUserId;

            .stream()
            .map(user -> BeanUtil.copyProperties(user, UserDTO.class))
            .collect(Collectors.toList());
    
    return Result.ok(users);
}
```

## 5. Kafka引入原因详解

### 5.1 当前架构分析

```
用户下单流程（项目当前设计）：

1. 用户请求 → 网关 → 业务服务
2. 业务服务 → Redis Stream (stream.orders)
3. Redis Stream → Relay (RedisStreamToKafkaRelay)
4. Relay → Kafka (voucher-order topic)
5. Kafka Consumer → 数据库
```

### 5.2 为什么有了Redis Stream还要Kafka

#### 5.2.1 Redis Stream的局限性

| 特性 | Redis Stream | 说明 |
|-----|------------|------|
| 持久化 | 内存优先 | 数据在内存，重启丢失 |
| 吞吐量 | 一般 | 单机约10万/秒 |
| 分区 | 不支持 | 无法水平扩展 |
| 顺序 | 部分保证 | 同一消费者组内 |
| 消费者组 | 基本支持 | 功能有限 |
| 消息回溯 | 弱 | 仅限于pending list |
| 生态 | 无 | 无法对接其他系统 |

#### 5.2.2 Kafka的优势

| 特性 | Kafka | 说明 |
|-----|-------|------|
| 持久化 | 磁盘存储 | 无限保留，可配置 |
| 吞吐量 | 极高 | 百万/秒 |
| 分区 | 支持 | 水平扩展 |
| 顺序 | 强保证 | 同分区保证顺序 |
| 消费者组 | 成熟 | 负载均衡 |
| 消息回溯 | 强 | 从任意offset消费 |
| 生态 | 丰富 | Flink, Spark等 |

#### 5.2.3 业务场景需求

秒杀系统的特点：
1. **瞬时高并发**：秒杀开始时，10万/秒的请求
2. **削峰填谷**：不能所有请求直接打给数据库
3. **持久化**：订单不能丢失
4. **重试机制**：订单创建可能失败，需要重试
5. **可观测**：需要监控消费积压

Redis Stream更适合：
- 即时通讯
- 简单异步任务
- 低延迟场景

Kafka更适合：
- 大数据管道
- 高吞吐场景
- 需要持久化的场景

### 5.3 项目中的中转设计

项目中使用了 **Redis Stream → Kafka** 的中转架构：

`RedisStreamToKafkaRelay.java`：

```java
@Component
public class RedisStreamToKafkaRelay implements CommandLineRunner {
    
    private static final String STREAM_KEY = "stream.orders";
    private static final String GROUP = "relay-g1";
    private static final String CONSUMER = "relay-c1";
    
    @Override
    public void run(String... args) {
        // 1.初始化消费者组
        initGroupIfNeeded();
        // 2.启动中转循环
        relayExecutor.submit(this::relayLoop);
    }
    
    private void relayLoop() {
        while (running) {
            try {
                // 1.从Redis Stream读取消息
                List<MapRecord<String, Object, Object>> records = 
                    stringRedisTemplate.opsForStream().read(...);
                
                // 2.转发到Kafka
                for (MapRecord record : records) {
                    forwardToKafka(record);
                }
                
                // 3.ACK消息
                stringRedisTemplate.opsForStream()
                    .acknowledge(STREAM_KEY, GROUP, record.getId());
                    
            } catch (Exception e) {
                log.error("Relay failed", e);
            }
        }
    }
}
```

### 6.1 核心角色图解

```
                    ┌─────────────────────────────────────┐
                    │           Kafka Cluster              │
                    │                                     │
┌─────────┐         │  ┌─────────┐  ┌─────────┐        │
│Producer │ ──────► │  │Broker 0 │  │Broker 1 │        │
└─────────┘         │  │(Leader) │  │(Follower)│        │
                    │  │  P0    │  │ P0      │        │
                    │  └─────────┘  └─────────┘        │
                    │                                     │
                    └─────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────────────────────────┐
                    │         Consumer Group               │
                    │                                     │
┌─────────┐        │  ┌─────────┐  ┌─────────┐         │
│Consumer1│ ◄────── │  │ Partition│  │Partition│        │
└─────────┘        │  │    0    │  │    1   │         │
                   │  └─────────┘  └─────────┘         │
                   └─────────────────────────────────────┘
```

### 6.2 核心概念解释

#### Broker（经纪人）

```
定义：Kafka服务实例，一个Kafka进程就是一个Broker
作用：负责接收 Producer 发送的消息，存储到磁盘
     响应 Consumer 的拉取请求
     
生产建议：
  - 单机单Broker（开发测试）
  - 多机多Broker（生产环境）3-5个
  - Broker数量 ≥ Partition数量
```

#### Topic（主题）

```
定义：消息的逻辑分类，类似数据库的表
特性：
  - Topic是发布订阅的核心
  - Producer向Topic发送消息
  - Consumer从Topic拉取消息
  - 一个Topic可以有多个Partition
  
示例：
  - voucher-order（订单Topic）
  - seckill-stock（库存Topic）
  - user-action（用户行为Topic）
```

#### Partition（分区）

```
定义：Topic的物理存储单元
特性：
  - 每个Partition是一个有序队列
  - 一个Partition只能被一个Consumer消费
  - Partition可设置副本（Replica）
  
图示：
  voucher-order Topic
  ├── Partition 0 ── [msg1, msg2, msg3, ...]
  ├── Partition 1 ── [msg4, msg5, msg6, ...]
  └── Partition 2 ── [msg7, msg8, msg9, ...]
```

#### Replica（副本）

```
定义：Partition的副本，用于高可用
类型：
  - Leader（主副本）：处理所有请求
  - Follower（从副本）：同步数据
  - ISR（In-Sync Replicas）：同步中的副本
  
failover过程：
  当Leader所在Broker宕机：
  1. Controller检��到Broker宕机
  2. 从ISR中选择新的Leader
  3. 其他Follower同步新Leader数据
```

#### Producer（生产者）

```
定义：向Kafka发送消息的客户端
职责：
  - 选择Topic和Partition
  - 序列化消息
  - 压缩消息
  - 发送确认
  
分区策略：
  1. 指定分区 → 直接发送
  2. Key分区 → Hash(key) % partition数
  3. 轮询 → 依次发送到各分区
```

#### Consumer（消费者）

```
定义：从Kafka拉取消息的客户端
消费方式：
  - 拉取模式（Pull）
  - 自动提交offset
  - 手动提交offset
  
消费者组：
  - 同一个组内的Consumer不会重复消费
  - 不同组可独立消费同一消息
```

#### Consumer Group（消费者组）

```
定义：一组消费者，共同消费一个Topic
特性：
  - 每个Partition只被组内一个Consumer消费
  - 组内负载均衡
  - 不同组互不影响
  
图示：
  Topic: voucher-order (3个分区)
  
  Group: order-group
  ├── Consumer1 → Partition 0
  ├── Consumer2 → Partition 1
  └── Consumer3 → Partition 2
```

### 6.3 消息语义详解

在分布式系统中，消息传递有三种语义：

#### 6.3.1 at-most-once（至多一次）

```
含义：消息可能丢失，但不会重复消费
场景：对实时性要求高，允许丢失的数据

发生场景：
  - Consumer获取消息后处理失败
  - offset已提交，但处理未完成
  - 重启后又消费了下一条

代码示例（不推荐）：
  // 自动提交offset（在拉取时）
  properties.put("enable.auto.commit", "true");
  
问题：
  如果处理失败，消息已消费，不会重试
```

#### 6.3.2 at-least-once（至少一次）

```
含义：消息不会丢失，但可能重复消费
场景：大多数业务场景（项目当前使用）

发生场景：
  - Consumer获取消息，处理成功
  - 但offset提交失败
  - 重启后会重复消费

代码示例（当前项目）：
  // 手动提交offset
  properties.put("enable.auto.commit", "false");
  
  // 处理完成后手动提交
  @KafkaListener
  public void consume(ConsumerRecord<String, String> record) {
      process(record);           // 处理消息
      listener.acknowledge();  // 手动提交offset
  }
```

#### 6.3.3 exactly-once（精确一次）

```
含义：消息既不丢失也不重复
场景：金融、计费等严格要求

实现方式：
  1. 幂等性（Idempotence）：
     - enable.idempotence=true
     - Producer增加PID+Partition+SeqID
     - Broker去重
  
  2. 事务（Transaction）：
     - 开启事务
     - 多消息原子性
     - 回滚机制

代码示例：
  // 生产者端开启幂等
  props.put("enable.idempotence", true);
  
  // 或使用事务
  kafkaTemplate.executeInTransaction(...)
```

### 6.4 项目中的Kafka配置

`KafkaConfig.java:48-61`：

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> 
        kafkaListenerContainerFactory(...) {
    
    // 1.创建工厂
    ConcurrentKafkaListenerContainerFactory<String, String> factory = 
        new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.setConcurrency(1);
    
    // 2.手动立即ACK
    // 收到消息后立即ACK，不等待批量处理
    factory.getContainerProperties().setAckMode(
        ContainerProperties.AckMode.MANUAL_IMMEDIATE
    );
    
    // 3.重试机制

    DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(
        kafkaTemplate,
        (record, ex) -> new TopicPartition(record.topic() + "-dlt", record.partition())
    );
    factory.setErrorHandler(
        new SeekToCurrentErrorHandler(recoverer, new FixedBackOff(1000L, 3L))
    );
    
    return factory;
}
```

### 6.5 重复消息问题解决方案

#### 6.5.1 问题分析

```
重复消息产生场景：

1. Producer发送成功，但ACK超时
   Producer                    Broker
      │                         │
      ├─── send(msg1) ──────►  │
      │                    [处理成功]
      │◄─── ACK [超时] ────────│
      │                         │
      ├─── retry(msg1) ──────►│  ← 重复发送！
      │                    [已存在]
```

#### 6.5.2 解决方案

| 方案 | 说明 | 适用场景 |
|-----|------|---------|
| **幂等性** | 生产者端开启去重 | 推荐 |
| **去重表** | 数据库唯一键 | 需要DB配合 |
| **消费端幂等** | 处理逻辑幂等 | 最常用 |
| **分布式锁** | 处理时加锁 | 并发处理 |

#### 6.5.3 项目当前处理

1. **SeekToCurrent重试**：失败后重新消费
2. **死信队列**：重试3次后进入死信Topic
3. **数据库幂等**：订单ID唯一约束

## 7. 本地缓存设计详解

### 7.1 本地缓存技术选型

项目使用的是 **Caffeine**（Guava Cache的升级版）：

```java

    public Cache<Long, Shop> shopLocalCache() {

```

### 7.2 缓存内容分析

| 缓存名称 | 存储内容 | 数据量 | TTL | 访问频率 |
|---------|---------|-------|-----|---------|
| shopLocalCache | 商户详情 | 1000条 | 5分钟 | 高 |
| shopTypeLocalCache | 商户类型列表 | 100条 | 10分钟 | 中 |

#### 7.2.1 为什么存储这些数据

1. **访问频繁**
   - 首页推荐、商户列表页、商户详情页都需要商户信息
   - 商户类型是通用配置，几乎不变化

2. **数据stable**
   - 商户信息变化不频繁
   - 修改时有专门的缓存失效机制

3. **性能要求高**
   - 频繁查询同一批商户
   - 本地缓存O(1)读取，避免网络开销

### 7.3 缓存更新机制

#### 7.3.1 缓存失效流程

```
更新商户信息流程：

1. 更新数据库
   ShopMapper.updateById(shop)

2. 删除Redis缓存
   redisTemplate.delete("cache:shop:" + shopId)

3. 发布缓存失效消息
   cacheInvalidationPublisher.publishShopInvalidation(shopId)
   
   │
   ▼
   
4. Redis Pub/Sub 广播
   "cache:invalidate" → {cacheName: "shop", key: "123"}
   
   │
   ▼
   
5. 所有节点监听并删除本地缓存
   CacheInvalidationListener.onMessage()
   → cacheClient.deleteShopLocalCache(123)
```

#### 7.3.2 发布者实现

`CacheInvalidationPublisher.java`：

```java
@Component
public class CacheInvalidationPublisher {
    
    @Resource
    private StringRedisTemplate stringRedisTemplate;
    
    public void publishShopInvalidation(Long id) {
        // 构造消息
        CacheInvalidationMessage message = new CacheInvalidationMessage();
        message.setCacheName("shop");
        message.setKey(String.valueOf(id));
        
        // 序列化
        String payload = objectMapper.writeValueAsString(message);
        
        // 发布到Redis Pub/Sub
        stringRedisTemplate.convertAndSend("cache:invalidate", payload);
        
        log.info("Published cache invalidation: {}", payload);
    }
}
```

#### 7.3.3 订阅者实现

`CacheInvalidationListener.java`：

```java
@Component
public class CacheInvalidationListener implements MessageListener {
    
    @Override
    public void onMessage(Message message, byte[] pattern) {
        // 1.解析消息
        String body = new String(message.getBody());
        CacheInvalidationMessage msg = objectMapper.readValue(body, ...);
        
        // 2.判断缓存类型并删除
        if ("shop".equals(msg.getCacheName())) {
            // 删除本地缓存
            cacheClient.deleteShopLocalCache(Long.valueOf(msg.getKey()));
            log.info("Deleted local shop cache: {}", msg.getKey());
        }
        
        if ("shop-type".equals(msg.getCacheName())) {
            cacheClient.deleteShopTypeLocalCache(msg.getKey());
            log.info("Deleted local shop-type cache: {}", msg.getKey());
        }
    }
}
```

### 7.4 多节点本地缓存问题

#### 7.4.1 问题分析

```
场景：3节点部署

节点A                          节点B                          节点C
┌─────────────┐               ┌─────────────┐               ┌─────────────┐
│本地缓存    │               │本地缓存    │               │本地缓存    │
│shop:123=X  │               │shop:123=X  │               │shop:123=X  │
│shop:456=Y  │               │shop:456=Y  │               │shop:456=Y  │
└─────────────┘               └─────────────┘               └─────────────┘

用户A → 节点A → 修改shop:123 ← 数据库

节点A: 删除Redis + 广播失效
节点B: [未收到广播？缓存未更新！]
节点C: [未收到广播？缓存未更新！]
```

#### 7.4.2 解决方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|-----|------|------|------|-------|
| **广播失效** | Pub/Sub通知所有节点删除 | 实时一致 | 网络复杂 | 项目当前 |
| **不设本地缓存** | 只用Redis | 简单 | 性能略低 | 数据一致性要求高 |
| **短TTL** | 每分钟过期 | 简单 | 有延迟 | 简单场景 |
| **一致性哈希** | 同数据走同节点 | 无需广播 | 破坏无状态 | 不推荐 |

#### 7.4.3 项目当前方案

项目使用的是 **Redis Pub/Sub 广播机制**：

```
架构图：

                    ┌──────────────────┐
                    │   Redis         │
                    │                 │
┌────────┐         │  ┌────────────┐  │
│ 节点A  │ ──────►│  │ pub/sub  │◄──────┐
└────────┘         │  │channel  │       │
                  │  └────────┘       │
┌────────┐       │                 │       └────────┐
│ 节点B  │ ◄────┤                 ├─────► │ 节点C  │
└────────┘       │                 │       └────────┘
                 └──────────────────┘
更新时：
  节点A → Redis发布 → 节点B+C 订阅 → 删除本地缓存
```

```java
    String tokenKey = "login:token:" + token;
```

#### 10.3.4 点赞核心代码

```java
@Service
@Slf4j
public class BlogLikeService {
    
    private static final String LIKE_KEY = "blog:liked:";
    private static final String STAT_KEY = "blog:stat:";
    
    @Resource
    private StringRedisTemplate redisTemplate;
    
    /**
     * 点赞/取消点赞
     */
    public boolean likeBlog(Long blogId, Long userId) {
        String likeListKey = LIKE_KEY + blogId;
        String statKey = STAT_KEY + blogId;
        
        // 1.检查是否已点赞
        Boolean isMember = redisTemplate.opsForSet().isMember(likeListKey, userId.toString());
        
        if (Boolean.TRUE.equals(isMember)) {
            // ===== 取消点赞 =====
            
            // 1.从Set移除
            redisTemplate.opsForSet().remove(likeListKey, userId.toString());
            
            // 2.点赞数-1
            redisTemplate.opsForHash().increment(statKey, "liked", -1);
            
            // 3.更新数据库（异步）
            asyncService.updateBlogLikeCount(blogId, -1);
            
            return false;  // 返回未点赞
            
        } else {
            // ===== 点赞 =====
            
            // 1.添加到Set
            redisTemplate.opsForSet().add(likeListKey, userId.toString());
            
            // 2.点赞数+1
            redisTemplate.opsForHash().increment(statKey, "liked", 1);
            
            // 3.更新数据库（异步）
            asyncService.updateBlogLikeCount(blogId, 1);
            
            return true;  // 返回已点赞
        }
    }
    
    /**
     * 获取点赞数
     */
    public long getLikeCount(Long blogId) {
        Object liked = redisTemplate.opsForHash().get(STAT_KEY + blogId, "liked");
        return liked != null ? Long.parseLong(liked.toString()) : 0;
    }
    
    /**
     * 获取点赞列表（前5）
     */
    public List<Long> getTopLikers(Long blogId) {
        Set<String> top5 = redisTemplate.opsForZSet()
            .reverseRange(LIKE_KEY + blogId, 0, 4);
        
        if (top5 == null || top5.isEmpty()) {
            return Collections.emptyList();
        }
        
        return top5.stream()
            .map(Long::parseLong)
            .collect(Collectors.toList());
    }
}
```
