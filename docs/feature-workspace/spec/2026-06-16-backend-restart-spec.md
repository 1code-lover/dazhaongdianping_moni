## 一、问题背景

### 1.1 当前现状

当前本地开发流程通常为：

1. 停止已有 Java 进程
2. 执行 `mvn clean compile`
3. 执行 `Start-Process mvn spring-boot:run`
4. 立即继续后续测试或接口调用

该流程存在一个稳定性问题：`Start-Process` 只是提交启动请求，Spring Boot 应用真正完成启动仍需要一段时间。在应用依赖 MySQL、Redis、Kafka、Tomcat 初始化的情况下，启动耗时可能达到 30 到 60 秒。

### 1.2 现有问题

当前流程缺少“应用已就绪”的确认步骤，导致：

- 启动命令返回后，实际服务尚未监听 `8081`
- 后续接口测试、页面验证、自动化测试会过早执行
- 使用者主观感受为“命令卡住”或“应用异常”
- 每次修改代码后都需要重复人工判断启动是否完成

### 1.3 目标

为本地开发提供一个稳定的“重启并等待服务可用”的统一脚本，减少人工判断，并在应用真正启动成功后再执行后续动作。

## 二、需求范围

### 2.1 本次包含

- 停止旧的后端 Java 进程
- 执行 Maven 编译
- 后台启动 Spring Boot 应用
- 等待端口 `8081` 进入监听状态
- 通过日志关键字或 HTTP 健康检查确认应用完成启动
- 在超时时给出明确失败原因

### 2.2 本次不包含

- 修改业务接口逻辑
- 修改数据库结构
- 修改前端页面逻辑
- 接管 Nginx、Redis、MySQL、Kafka 的启动

---

## 三、业务流程设计

### 3.1 目标流程

```text
开发者执行重启脚本
    ->
识别并停止当前项目对应 Java 进程
    ->
执行 mvn clean compile
    ->
编译成功后后台启动 spring-boot:run
    ->
轮询检查 8081 端口
    ->
轮询检查日志关键字或健康接口
    ->
启动成功，返回可继续测试
```

### 3.2 异常流程

```text
编译失败
    -> 直接退出，并输出编译错误

端口长时间未监听
    -> 读取最新日志摘要并退出

监听成功但应用未完全初始化
    -> 继续等待关键日志或健康检查通过

超时
    -> 输出超时说明、日志位置、建议排查项
```

## 四、数据模型设计

本功能不新增数据库表。

为便于脚本内部处理，需要定义一个“启动状态模型”：

| 状态 | 说明 |
|------|------|
| `stopped` | 已停止旧进程 |
| `compiling` | 正在编译 |
| `starting` | 已提交启动命令，等待应用初始化 |
| `port_ready` | 端口已监听 |
| `app_ready` | 应用完成启动，可继续测试 |
| `failed` | 编译失败、启动失败或超时 |

该状态仅用于脚本运行期输出，不持久化到数据库。

## 五、接口设计

### 5.1 脚本入口设计

新增 PowerShell 脚本，例如：

```powershell
scripts/restart-backend.ps1
```

### 5.2 输入参数设计

建议支持以下参数：

| 参数 | 是否必填 | 默认值 | 说明 |
|------|------|------|------|
| `Port` | 否 | `8081` | 后端监听端口 |
| `StartupTimeoutSec` | 否 | `90` | 启动超时时间 |
| `CompileFirst` | 否 | `true` | 是否先执行编译 |
| `LogFile` | 否 | `app.log` | 标准输出日志文件 |
| `ErrorLogFile` | 否 | `app-error.log` | 标准错误日志文件 |

### 5.3 输出设计

脚本执行后需要输出清晰结果：

- 编译是否成功
- 新启动进程 PID
- 当前监听端口
- 是否检测到启动成功关键日志
- 若失败，输出失败阶段和日志文件位置

### 5.4 成功判定

满足以下条件之一即可视为应用启动成功：

1. `8081` 端口开始监听，且日志中出现 `Started HmDianPingApplication`
2. `8081` 端口开始监听，且健康检查接口返回成功

优先采用“端口监听 + 启动日志”组合，避免仅监听端口但 Spring 上下文仍未完全就绪。

## 六、方案要点

### 6.1 进程停止策略

不能再使用无差别的 `Get-Process java | Stop-Process`，原因如下：

- 会误杀 IDE、其他 Java 服务、测试进程
- 难以区分当前项目与其他项目

应改为优先识别“当前项目目录对应的 Java 进程”或“当前目标端口对应的进程”，只停止目标进程。

### 6.2 启动等待策略

不能使用固定等待 60 秒作为唯一手段，原因如下：

- 机器负载变化会导致启动耗时不同
- 固定等待过短会误判，过长会浪费时间

建议改为“轮询等待”：

1. 每隔 2 秒检测一次端口状态
2. 每隔 2 秒检测一次日志关键字
3. 在超时时间内成功即提前返回

### 6.3 日志处理策略

脚本启动前应清理或归档旧日志，避免把上一次启动成功的日志误判为本次成功。

---

## 七、验收标准

满足以下条件视为需求完成：

1. 执行一次脚本即可完成停止、编译、启动、等待四个动作
2. 应用未启动完成前，脚本不会误报成功
3. 应用启动成功后，脚本明确提示可以继续测试
4. 编译失败、启动超时、端口占用时，脚本能输出可定位问题的信息
5. 不再误杀无关 Java 进程

---

## 八、风险与注意事项

### 8.1 已知风险

- 本机若同时运行多个 Maven 或 Java 项目，进程识别逻辑需要足够保守
- 若应用依赖的 MySQL、Redis、Kafka 未启动，后端可能永远无法就绪
- 若日志文件未轮换，关键字匹配可能误命中历史记录

### 8.2 规避措施

- 优先按端口或工作目录识别目标进程
- 脚本启动前打印依赖检查结果
- 启动前清空本次使用的日志文件

---

## 九、建议实施结论

建议新增一个统一的本地重启脚本，替代手工拼接命令。

该脚本的核心不是“多等 60 秒”，而是：

- 精准停止目标进程
- 后台启动应用
- 轮询等待端口与日志双重就绪
- 明确输出成功或失败结论

这样可以稳定解决“启动命令返回太快，后续步骤误以为卡住或误判失败”的问题。
