# 项目整改计划

> **说明**：这是一份面向当前 `hm-dianping` 项目的中文整改计划，重点围绕“正确性风险、并发安全、依赖治理、工程卫生、仓库整理”五个方向展开。可按优先级分阶段执行。

**目标：** 先消除会导致数据不一致或并发错误的高风险问题，再逐步提升代码可维护性、仓库整洁度和后续迭代效率。

**适用范围：** 当前工作区 `D:\Java\hm-dianping`

**计划日期：** 2026-05-02

---

## 一、整改目标

本次整改的目标分为三层：

1. 修复高风险问题，避免出现库存扣减错误、订单数据不一致、分布式锁误删等影响业务正确性的情况。
2. 收敛技术债，降低依赖混乱、注释乱码、无关代码混入主工程带来的维护成本。
3. 优化工程卫生，整理仓库结构、忽略无关产物，让项目更适合长期开发、演示和对外展示。

---

## 二、问题分级

### P0：必须优先处理

这类问题已经影响系统正确性或并发安全，建议最先处理。

1. 秒杀订单落库缺少明确事务边界
2. `CacheClient` 中的互斥锁释放逻辑存在并发风险
3. Redis Stream 到 Kafka 的中转消费者名称写死，不适合多实例部署

### P1：应尽快处理

这类问题会明显拉高维护成本，建议在 P0 完成后紧接着处理。

1. `pom.xml` 中存在冗余依赖与版本管理不统一问题
2. 主工程目录混入演示类、算法练习类
3. 核心代码注释和文档存在乱码

### P2：建议收尾处理

这类问题不一定立刻影响功能，但会影响仓库整洁度和项目观感。

1. `.gitignore` 规则仍不完整
2. 报告、面试稿、压测资料混放在仓库根目录
3. 示例配置文件还可以继续脱敏

## 三、详细整改任务

### 任务 1：修复秒杀订单事务一致性问题

**优先级：** P0

**问题描述：**  
当前秒杀下单链路中，库存扣减和订单保存发生在同一业务流程里，但没有形成清晰、可靠的事务边界。这样会导致一种风险：库存已经扣减成功，但订单保存失败，最终出现“库存少了、订单没了”的数据不一致。

**涉及文件：**

- [VoucherOrderServiceImpl.java](/D:/Java/hm-dianping/src/main/java/com/hmdp/service/impl/VoucherOrderServiceImpl.java:75)

**整改动作：**

1. 明确“扣库存 + 保存订单”的事务边界。
2. 检查 `createVoucherOrder()` 是否通过 Spring 事务代理执行。
3. 避免在事务外先扣库存、事务内再保存订单的分裂写法。
4. 梳理异常分类，明确哪些异常要回滚，哪些异常作为幂等忽略。
5. 必要时增加集成测试，覆盖库存扣减成功但订单保存失败的场景。

**验收标准：**

1. 出现异常时，库存和订单数据不会出现不一致。
2. 重复消费、唯一索引冲突等幂等场景行为符合预期。
3. 至少有一条测试用例覆盖事务回滚场景。

### 任务 2：修复 `CacheClient` 中的锁实现问题

**优先级：** P0

**问题描述：**  
`queryWithMutex()` 当前存在两个典型并发问题：

1. 没拿到锁的线程也会进入 `finally` 执行 `unlock()`。
2. `unlock()` 只是简单删除 key，没有校验“锁是不是自己加的”。

这会导致误删别的线程持有的锁，从而让缓存重建逻辑在高并发下失去保护。

**涉及文件：**

- [CacheClient.java](/D:/Java/hm-dianping/src/main/java/com/hmdp/utils/CacheClient.java:334)
- [SimpleRedisLock.java](/D:/Java/hm-dianping/src/main/java/com/hmdp/utils/SimpleRedisLock.java:1)
- [unlock.lua](/D:/Java/hm-dianping/src/main/resources/unlock.lua:1)

**整改动作：**

1. 修改 `queryWithMutex()`，确保只有成功加锁的线程才会释放锁。
2. 为锁增加 owner 标识，而不是只写固定值 `"1"`。
3. 释放锁时采用 Lua 脚本或等价原子校验方案，确保“比较 owner + 删除 key”是原子操作。
4. 评估是否复用已有的 `SimpleRedisLock`，避免维护两套风格不一致的锁逻辑。
5. 增加并发测试或至少补充单元测试，覆盖“未持锁线程释放锁”的错误场景。

**验收标准：**

1. 未拿到锁的线程不会删掉别人的锁。
2. 只有持锁线程可以释放锁。
3. 缓存重建逻辑在并发条件下不再出现明显竞态。

### 任务 3：改造 Redis Stream Relay 的消费者标识

**优先级：** P0

**问题描述：**  
`RedisStreamToKafkaRelay` 中的消费者名固定写成了 `relay-c1`。单机运行时问题不大，但一旦多实例部署，不同节点会共用同一个消费者身份，Pending List 的恢复和消息归属都会变得混乱。

**涉及文件：**

- [RedisStreamToKafkaRelay.java](/D:/Java/hm-dianping/src/main/java/com/hmdp/mq/RedisStreamToKafkaRelay.java:52)
- [application.yaml](/D:/Java/hm-dianping/src/main/resources/application.yaml:1)

**整改动作：**

1. 将消费者名改为实例级唯一值。
2. 唯一值可来自主机名、进程号、时间戳、UUID 或配置项组合。
3. 将 group 和 consumer 的命名策略写入配置或注释说明。
4. 检查 Pending List 恢复逻辑在多实例下是否仍符合预期。
5. 如有需要，补充一段 README 说明当前中转器适合的部署方式。

**验收标准：**

1. 多实例启动时不会共用同一个 consumer name。
2. Pending List 的恢复语义清晰，不会互相抢占同名消费者遗留状态。

### 任务 4：收敛 `pom.xml` 冗余依赖

**优先级：** P1

**问题描述：**  
当前项目同时保留了 Jackson、`json-lib`、`fastjson` 等多套 JSON 方案，而实际主链路主要使用 Jackson。同时，Redis 相关依赖还存在手动排除后再手动指定版本的情况，增加了兼容性风险和维护复杂度。

**涉及文件：**

- [pom.xml](/D:/Java/hm-dianping/pom.xml:19)

**整改动作：**

1. 搜索项目里是否真实使用了 `json-lib` 和 `fastjson`。
2. 未使用则移除，统一保留 Jackson 作为主序列化方案。
3. 梳理 Spring Boot 管理下的 Redis 依赖，评估是否回归 BOM 默认版本。
4. 检查删除冗余依赖后编译、测试是否正常。

**验收标准：**

1. `pom.xml` 中不再保留无实际用途的 JSON 依赖。
2. Redis 相关依赖版本来源清晰，不再出现不必要的手动覆盖。
3. `mvn test` 或最小编译验证通过。

## 六、验收方式建议

为了避免“改了很多，但没法确认效果”，建议每一类整改都附带最小验证动作：

1. 事务和并发相关整改：
   - 运行对应单元测试或集成测试
   - 人工制造异常验证是否会回滚
2. 依赖整理：
   - 执行一次最小编译验证
   - 检查是否仍有未使用依赖引用
3. 文档和仓库整理：
   - 查看 `git status`
   - 打开 README 和核心源码确认可读性

## 八、后续建议

如果准备继续推进，我建议下一步不要同时开很多整改项，而是按下面节奏走：

1. 先落地 P0 三项整改
2. 每完成一项就跑一次最小验证
3. P0 稳定后，再统一处理 P1 和 P2

这样节奏最稳，也最容易看出每次修改带来的收益。
