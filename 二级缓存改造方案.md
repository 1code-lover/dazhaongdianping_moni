# hm-dianping 二级缓存改造方案

## 1. 文档目的

本方案用于指导 `hm-dianping` 项目第一版二级缓存改造落地，目标是在**不扩大改造范围、不影响现有核心业务链路**的前提下，为热点读接口引入 `L1 本地缓存 + L2 Redis` 结构，并配合后续压测验证收益。

这份文档关注的是：

- 改哪些类
- 为什么这样改
- 查询链路怎么走
- 更新链路怎么失效
- 第一版改造边界是什么

这份文档**不是**压测报告，也**不是**最终上线方案。当前目标是先完成一个结构清晰、风险可控、便于讲述与验证的第一版实现。

## 2. 当前项目现状

结合当前代码，项目并不是“没有缓存”，而是“已经有 Redis 单层缓存，但还没有二级缓存”。

当前已具备的基础：

- `GET /shop/{id}` 已使用 Redis 缓存
- `GET /shop-type/list` 已使用 Redis 缓存
- 已存在通用缓存工具类 `CacheClient`
- `CacheClient` 已支持：
  - 缓存穿透保护
  - 互斥锁重建缓存
  - 逻辑过期方案
- 店铺更新后，当前会删除 Redis 中对应的 `shop` 缓存

当前存在的问题：

- 只有 Redis 一层缓存，热点请求仍会直接打到 Redis
- `Shop` 与 `ShopType` 的缓存实现方式不统一
- 本地缓存 `L1` 尚未接入
- 更新后的缓存失效目前只覆盖 Redis，没有覆盖本地缓存

## 3. 第一版改造目标

第一版只追求三件事：

1. 为热点读接口增加本地缓存层，降低 Redis 压力
2. 统一热点接口的缓存访问方式，减少重复逻辑
3. 保证更新后缓存能够正确失效，避免明显脏读风险

本次改造完成后，希望在压测中看到：

- `GET /shop/{id}` 的 TP95 明显下降
- `GET /shop-type/list` 的 TP95 明显下降
- Redis OPS 下降
- 错误率不升高

## 4. 改造边界

### 4.1 本次要改的范围

- `GET /shop/{id}`
- `GET /shop-type/list`
- 与上述两个接口直接相关的缓存公共组件

### 4.2 本次不改的范围

以下内容先不纳入第一版：

- `GET /shop/of/type` 分页列表缓存
- 秒杀链路
- Lua 脚本
- 登录链路缓存
- 多实例节点间本地缓存同步

原因很简单：第一版目标是先把收益最明显、实现最稳的场景做扎实，避免一开始把范围铺太大导致方案发散。

## 5. 方案选型

采用如下缓存结构：

- `L1`：Caffeine 本地缓存
- `L2`：Redis 缓存
- `DB`：MySQL

设计原则：

- 读优先命中本地缓存，减少跨进程网络开销
- L1 未命中再查 Redis
- Redis 未命中再查数据库
- 数据库回源成功后，同时回填 Redis 和 L1
- 更新操作后，同时删除 Redis 与本地缓存

## 6. 查询链路设计

### 6.1 `shop/{id}` 查询链路

查询顺序：

1. 先查 L1 Caffeine
2. L1 未命中则查 Redis
3. Redis 命中则回填 L1 并返回
4. Redis 未命中则查数据库
5. 数据库命中后写入 Redis，并同步写入 L1
6. 数据库未命中则写入空值保护，防止缓存穿透

说明：

- `shop/{id}` 是典型热点读接口，最适合优先上二级缓存
- 这个接口已有 `CacheClient` 基础，可优先在工具层抽象出统一实现

### 6.2 `shop-type/list` 查询链路

查询顺序与 `shop/{id}` 一致：

1. 查 L1
2. 查 Redis
3. 查数据库
4. 回填 L2 与 L1

说明：

- `shop-type/list` 数据变化频率低、数据量小、命中率通常高
- 这是二级缓存收益最稳定、实现最简单的场景之一

## 7. 更新与失效链路设计

### 7.1 `shop` 更新链路

当前 `shop` 更新已经具备“更新数据库后删除 Redis”的基础，改造后建议统一为：

1. 更新数据库
2. 删除 Redis 中对应 key
3. 删除本地缓存中对应 key

这样做的目的：

- 避免更新后本地缓存继续返回旧值
- 保持失效动作简单直接
- 第一版不引入过多复杂一致性机制

### 7.2 `shop-type` 更新链路

如果后续存在 `shop-type` 变更操作，也应采用同样策略：

1. 更新数据库
2. 删除 Redis 中 `shop-type:list`
3. 删除本地缓存中 `shop-type:list`

如果当前项目没有实际的 `shop-type` 更新接口，这部分可以先把失效方法准备好，不强行补业务入口。

## 8. TTL 与缓存策略建议

第一版建议采用“L1 较短、L2 较长”的策略：

- `shop`：
  - L1 TTL：5 分钟
  - Redis TTL：沿用当前配置 30 分钟
- `shop-type:list`：
  - L1 TTL：10 分钟
  - Redis TTL：沿用当前配置 30 分钟

说明：

- L1 过期时间短于 Redis，有助于控制本地脏数据停留时间
- 第一版不建议把 TTL 设计得过于复杂，先保证简单可解释

空值缓存建议：

- Redis 继续保留空值缓存机制
- L1 第一版不缓存空值，避免本地空值状态传播过久

## 9. 类改造计划

### 9.1 必改类

#### `src/main/java/com/hmdp/service/impl/ShopServiceImpl.java`

职责：

- 接入 `shop/{id}` 的二级缓存查询
- 在 `update(Shop shop)` 中补充本地缓存失效

建议改动：

- 将当前 `queryById()` 的缓存查询改为统一的 L1 -> L2 -> DB
- 保留现有数据库更新逻辑
- 在删除 Redis key 的同时删除本地缓存 key

#### `src/main/java/com/hmdp/service/impl/ShopTypeServiceImpl.java`

职责：

- 将当前手写 Redis 缓存逻辑升级为二级缓存逻辑
- 尽量与 `ShopServiceImpl` 走统一缓存访问模式

建议改动：

- 去掉散落在业务类中的重复缓存判断逻辑
- 改为调用统一缓存组件

#### `src/main/java/com/hmdp/utils/CacheClient.java`

职责：

- 承担公共缓存访问逻辑
- 统一封装 `L1 -> L2 -> DB` 查询流程

建议改动：

- 增加面向二级缓存的公共查询方法
- 保留当前已有的 Redis 单层缓存方法，避免影响已有代码
- 提供缓存删除能力，供业务更新时调用

#### `src/main/java/com/hmdp/utils/RedisConstants.java`

职责：

- 集中维护缓存相关常量

建议改动：

- 增加本地缓存相关常量
- 增加本地缓存 TTL 配置常量
- 如有需要，补充缓存名称常量

### 9.2 新增类

#### `src/main/java/com/hmdp/config/LocalCacheConfig.java`

职责：

- 配置 Caffeine CacheManager 或相关 Bean

建议内容：

- 统一定义本地缓存
- 控制最大容量
- 控制过期策略

如果你希望后续扩展更多本地缓存场景，这个配置类会非常有价值。

#### 可选：`src/main/java/com/hmdp/utils/LocalCacheNames.java`

职责：

- 统一管理本地缓存名称字符串

是否必须新增：

- 不是必须
- 如果第一版只做两个缓存，也可以先放到常量类里

## 10. 关键设计取舍

### 10.1 为什么第一版只改两个接口

因为这两个接口最适合做收益验证：

- 热点明显
- 读多写少
- 一致性要求相对可控
- 压测结果容易体现改造成果

### 10.2 为什么不先改分页列表

分页列表的问题更复杂：

- 参数组合更多
- key 设计更复杂
- 命中率不一定高
- 一旦缓存设计不当，收益可能不明显

所以第一版不建议把它和热点详情接口混在一起做。

### 10.3 为什么不先做多实例本地缓存同步

因为第一版目标是先证明单机二级缓存方案有效。

多实例同步一旦引入，通常要继续考虑：

- 消息广播
- 主动通知失效
- 版本号比对
- 更复杂的一致性保证

这会显著增加实现和讲解成本，不适合第一步就上。

## 11. 实施顺序建议

建议按以下顺序落地：

1. 引入 Caffeine 依赖
2. 新增本地缓存配置类
3. 扩展 `CacheClient`，支持二级缓存公共查询
4. 改造 `ShopServiceImpl`
5. 改造 `ShopTypeServiceImpl`
6. 补充缓存失效方法
7. 本地联调验证
8. 再执行基线压测与改造后压测

## 12. 联调验证建议

改造完成后，至少验证以下内容：

- 第一次访问 `shop/{id}` 时能够正常回源并写入缓存
- 第二次访问能够命中本地缓存
- 更新 `shop` 后再次读取时返回新值
- `shop-type/list` 能正确命中本地缓存
- Redis 不可用时，系统行为是否符合预期

注意：

- 第一版优先保证正确性
- 性能收益通过后续压测来证明
- 不要在还没验证链路正确前，就过早引入复杂优化

## 13. 最终建议

本项目的二级缓存改造，第一版最合适的做法不是“大而全”，而是“先把最值钱的两个热点读接口做扎实”。

最推荐的落地方式是：

- 只做 `shop/{id}` 和 `shop-type/list`
- 用 `Caffeine + Redis`
- 工具层统一查询逻辑
- 更新后统一删除 L1 与 L2
- 暂不处理多实例同步

这样做的优点是：

- 风险低
- 改动面可控
- 结果容易压测验证
- 方案容易在答辩、面试、复盘中讲清楚

后续如果第一版效果理想，再继续扩展到：

- 分页列表缓存
- 热点预热
- 多实例本地缓存同步
- 更细粒度的缓存治理
