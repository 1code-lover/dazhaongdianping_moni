# hm-dianping 二级缓存改造清单（L1 本地缓存 + L2 Redis）

## 1. 改造目标

- 在现有 Redis 缓存基础上，引入 JVM 本地缓存（L1），形成 `L1 -> L2 -> DB` 的分层读取路径。
- 降低热点接口的平均响应时间和 Redis QPS，提升系统在高并发场景下的稳定性。
- 构建可演示、可量化、可回滚的工程化改造方案，用于面试中的项目亮点输出。

## 2. 改造范围（建议先小后大）

- 第一阶段仅覆盖高频读接口：
  - `ShopServiceImpl.queryById`
  - `ShopTypeServiceImpl.queryTypeListWithCache`
- 暂不覆盖秒杀库存、订单流等强一致链路（这些更适合继续走 Redis/Lua 原子操作）。

## 3. 目标架构与读写流程

## 3.1 读取流程

1. 先查 L1（本地缓存，Caffeine）。
2. L1 miss 后查 L2（Redis）。
3. L2 miss 后查 DB，并回填 L2 与 L1。
4. 对空值进行短 TTL 缓存，防止缓存穿透。

## 3.2 写入/更新流程

1. 先更新 DB（事务内）。
2. 删除 L2 key（Redis）。
3. 发布失效消息（Redis Pub/Sub 或 Stream）。
4. 各实例订阅消息并删除本机 L1 key。

> 说明：单机下可直接删本地缓存；多实例下必须做失效广播，否则会出现节点间脏读。

## 4. 分阶段改造清单

## 阶段 A：基础能力建设（本周可完成）

- [ ] 引入 `Caffeine` 依赖，并新增本地缓存配置类（容量、过期时间、统计开关）。
- [ ] 抽象 `TwoLevelCacheClient`（统一封装 L1/L2 查询模板），避免业务代码重复。
- [ ] 支持缓存空值占位与短 TTL，避免穿透流量直接打 DB。
- [ ] 增加缓存 key 规范（统一前缀、版本号、业务域），防止冲突。

验收标准：
- [ ] 本地压测下，热点 key 二次请求命中 L1。
- [ ] Redis QPS 相比改造前下降（记录基线和结果）。

## 阶段 B：业务落地（优先 Shop 与 ShopType）

- [ ] 将 `ShopServiceImpl.queryById` 接入二级缓存模板。
- [ ] 将 `ShopTypeServiceImpl` 从手写 Redis 逻辑迁移到统一模板。
- [ ] 对 `updateShop` 或相关更新接口补齐 L2 删除与 L1 失效。
- [ ] 明确 TTL 策略：
  - `Shop`：中等 TTL + 抖动（防雪崩）
  - `ShopType`：较长 TTL（低变更数据）

验收标准：
- [ ] 功能回归通过（查询、更新后读取一致）。
- [ ] 热点查询 TP95 有明显改善（建议目标 20%+）。

## 阶段 C：多实例一致性与防护增强

- [ ] 增加 L1 失效广播（Pub/Sub 或 Stream 均可）。
- [ ] 为热点 key 增加互斥重建或逻辑过期，防止击穿。
- [ ] 加入限流降级策略：Redis 故障时可短时读 L1 兜底（仅弱一致读场景）。
- [ ] 设置开关：可一键关闭 L1，回退到 Redis 单层缓存。

验收标准：
- [ ] 两个应用实例下，更新后跨节点数据可在秒级收敛。
- [ ] 压测中无明显缓存抖动或 DB 峰值突刺。

## 阶段 D：可观测性与工程化闭环

- [ ] 暴露缓存指标：L1 hit/miss、L2 hit/miss、回源 DB 次数、平均加载耗时。
- [ ] 记录关键日志：缓存重建、批量失效、异常回源。
- [ ] 补充测试：
  - 单元测试：缓存命中与失效逻辑
  - 集成测试：更新后跨层一致性
  - 并发测试：热点 key 下击穿保护
- [ ] 输出改造对比报告（前后延迟/QPS/错误率）。

验收标准：
- [ ] 指标可在日志或监控页面看到。
- [ ] CI 中至少包含核心缓存测试用例。

## 5. 参数建议（首版保守值）

- L1（Caffeine）
  - `maximumSize`: 10_000（按内存再调）
  - `expireAfterWrite`: 30s~120s（按业务区分）
  - 开启 `recordStats`（便于展示命中率）
- L2（Redis）
  - 保持现有 TTL，并增加随机抖动（如 `+ 0~10%`）
  - 空值 TTL 建议短期（如 1~2 分钟）

## 6. 风险与规避

- 本地缓存数据陈旧：通过失效广播 + 短 TTL 规避。
- 缓存雪崩：TTL 抖动 + 预热 + 限流。
- 多级缓存复杂度上升：统一 `TwoLevelCacheClient`，避免业务散落实现。
- 过度缓存导致内存压力：按容量上限与对象大小评估，压测后再放量。

## 7. 面试表达模板（可直接复用）

- 我将项目从单层 Redis 缓存升级为二级缓存架构，读取路径是 `Caffeine -> Redis -> MySQL`。
- 热点查询优先走 L1，显著降低了 Redis 压力，并通过 TTL 抖动和空值缓存解决雪崩与穿透问题。
- 为了解决多实例本地缓存一致性，我增加了缓存失效广播机制，更新后可在秒级完成跨节点失效。
- 我把二级缓存能力抽成统一组件，减少业务重复代码，并补上命中率指标与压测对比，保证方案可观测、可验证、可回滚。
