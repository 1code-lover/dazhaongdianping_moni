# 唯一索引与补偿链路验证步骤

## 1. 唯一索引验证

执行 SQL：

```sql
ALTER TABLE tb_voucher_order
ADD CONSTRAINT uk_user_voucher UNIQUE (user_id, voucher_id);
```

验证方式：

1. 先让同一个用户对同一张券成功创建一笔订单
2. 再重复消费或重复插入相同 `(user_id, voucher_id)`
3. 预期：数据库命中唯一索引，应用日志出现：

```text
Create voucher order duplicate key ignored
```

说明数据库唯一索引已经成为最终幂等兜底。

## 2. 补偿/重试链路最小验证

当前项目已经具备两层最小恢复能力：

- Kafka Consumer 失败后不 ack，触发重试
- Redis Stream Relay 启动时恢复 pending-list

### 2.1 Kafka 重试验证

将：

```yaml
app:
  kafka:
    consumer:
      fail-once: true
```

启动后发一笔秒杀请求，预期日志：

1. `Kafka consume fail-once triggered`
2. `Kafka consume failed`
3. `Seeking to offset ...`
4. 同一 traceId 再次出现 `Kafka consume success`

这说明手动 ack + Kafka 重试生效。

### 2.2 Relay pending-list 恢复验证

验证步骤：

1. 关闭 Kafka broker
2. 发起秒杀请求，让消息进入 Redis Stream，但转发 Kafka 失败
3. 恢复 Kafka
4. 重启应用
5. 观察日志是否出现 pendingReplay=true 的：

```text
Relay forward success, ..., pendingReplay=true
```

如果出现，说明 Relay 的 pending-list 恢复生效。
