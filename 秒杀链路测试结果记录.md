本次测试主要用于验证当前秒杀链路在接入 `Redis Lua + Redis Stream + Kafka + MySQL` 后的基础表现，重点观察：

- ID 生成能力

本轮测试更偏向 **功能可用性 + 基线性能观察**，还不属于完整生产级压测。

## 4. 测试结果汇总

### 4.1 RedisIdWorker 基线测试

测试函数：

- `testIdWorker()`

测试输出：

```text
time = 1386
```

结合代码逻辑：

- 线程数：300
- 每线程生成 ID 数量：100
- 总请求数：30000

估算结果：

- 总耗时：1386 ms
- 估算 QPS：约 `30000 / 1.386 ≈ 21645/s`

#### 分析

这个结果说明：

1. `RedisIdWorker` 当前生成订单 ID 的能力是足够的
2. 3 万次并发 ID 生成在 1.386 秒内完成，说明它暂时不是秒杀链路的主要瓶颈
3. 对于当前项目规模，这个 ID 生成吞吐可以支撑秒杀入口的订单 ID 分配

#### 结论

- `RedisIdWorker` 当前表现正常
- 不是这轮秒杀链路测试中的性能短板

### 4.2 单用户高并发压测

测试函数：

- `benchmarkSeckillVoucherSingleUserContention()`

测试参数：

- `voucherId = 10`
- `threadCount = 500`
- `multiUser = false`
- 所有请求使用同一个用户 ID

测试输出：

```text
Benchmark init: voucherId=10, threadCount=500, multiUser=false
Benchmark ready: allReady=false, readyCount=200
====== Seckill Benchmark ======
voucherId = 10
threadCount = 500
multiUser = false
completed = true
started = 500
finished = 500
remaining = 0
costMs = 206
qps = 2427
success = 1
duplicate = 499
noStock = 0
error = 0
===============================
```

#### 结果解读

##### 1. 为什么 `readyCount=200`

这不是异常，也不是卡住。

原因是测试线程池大小设置为：

- `Math.min(threadCount, 200)`

也就是：

- 虽然总任务数是 500
- 但同时工作的线程池大小只有 200

所以在 `ready.await(10s)` 这个阶段，能立刻进入等待起跑状态的任务最多先看到 200 个，这属于测试线程池配置导致的正常现象。

##### 2. 为什么只有 1 个成功

因为这是 **单用户并发压测**，目标本来就是验证一人一单。

结果：

- `success = 1`
- `duplicate = 499`

说明：

- 第一个请求成功拿到下单资格
- 后面 499 个请求都被 Redis Lua / 业务判重链路拦截为重复下单

##### 3. 为什么 `noStock = 0`

因为当前库存足够，且重复下单拦截优先发生。

所以失败主要来自：

- 一人一单校验命中

而不是：

- 库存耗尽

##### 4. 吞吐结果

- 总耗时：206 ms
- 估算 QPS：2427

这个 QPS 更准确地说是：

- 当前测试线程模型下，单用户秒杀入口处理 500 次请求时的服务层基线吞吐

它不是完整 HTTP 压测 QPS，也不是最终对外公布的系统吞吐能力。

#### 结论

本轮测试说明：

1. 一人一单逻辑生效
2. 重复请求被稳定拦截
3. 没有出现异常错误
4. 当前服务层单用户高并发入口吞吐约为 2427 QPS

#### 面试表达建议

可以概括成：

`我做过单用户高并发压测，500 个并发请求下只有 1 个成功，其余 499 个都被一人一单逻辑拦截，没有出现错误，说明秒杀入口的重复下单控制是有效的。`

### 4.4 HTTP 层 JMeter 压测（多 token，端到端）

本节与 **4.2 / 4.3**（直接调 `VoucherOrderService` 的 SpringBootTest）互补：这里走 **`POST /voucher-order/seckill/{id}`**，经过 **Tomcat → Controller → Redis Lua**，更贴近真实接口。

#### 4.4.1 环境与工具

| 项 | 说明 |
|----|------|
| 部署 | **单机**：本机同时跑 Spring Boot、JMeter、Redis、MySQL、Kafka（若有） |
| 脚本 | 仓库内 `秒杀抢购.jmx` |
| 多用户 | `BatchTokenGeneratorTest` 生成 `tokens.txt`，**CSV Data Set Config** 每行一个 `authorization`；**Sharing mode** 建议 **All threads** 轮询 |
| 断言 | **响应断言（OR）**：响应体包含 `"success":true` **或** `不能重复下单` **或** `库存不足`，避免把业务拒绝当成 JMeter「失败」 |
| 说明 | 以下为**本地相对基线**，**不能**直接当作生产环境承诺值 |

更多命令与脚本说明见：`scripts/LOADTEST.md`。

#### 4.4.2 场景记录（对照用）

| 场景 | 线程 / Ramp-up / 循环 | 聚合报告要点（量级） | 说明 |
|------|-------------------------|----------------------|------|
| A 温和起压 | 约 1000 / **10s** / 1 | Throughput ~**100/s**，Error **0%**，延迟 ms 级 | 易稳定，适合基线 |
| B 长 Ramp-up | 3000 / **60s** / 1 | Throughput ~**50/s**，Error **0%** | 起线程慢，**峰值并发被摊薄**，吞吐偏低属正常 |
| C 尖峰 | 3000 / **1s** / 1 | 可能出现 **Error%**（连接类），Throughput 数值需结合失败类型解读 | 测瞬时冲击；失败快结束时 Throughput 会被「墙钟时间」放大 |
| **D 主结果（推荐面试引用）** | 3000 / **1s** / 1（与截图一致） | **#Samples 3000**，Throughput **~1910/s**，Error **0%** | 见下表 |

#### 4.4.3 主结果明细（场景 D，聚合报告）

以下为 **JMeter 5.6.3 聚合报告** 一屏数据（线程组名如「秒杀抢购一万人」等，以实际 jmx 为准）：

| 指标 | 值 |
|------|-----|
| # Samples | 3000 |
| Average | 342 ms |
| Median | 390 ms |
| 90% Line | 700 ms |
| 95% Line | 762 ms |
| 99% Line | 875 ms |
| Min / Max | 1 ms / 885 ms |
| Error % | 0.00% |
| **Throughput** | **1910.8 / sec** |

**解读要点**：

1. **Throughput（≈QPS）**：JMeter 为 **总样本数 ÷ 整段测试墙钟时间**；在 **0% 断言失败** 前提下，可表述为：本轮 **HTTP 秒杀接口**平均约 **1.9k 请求/秒** 的处理节奏（单机环境）。  
2. **延迟**：平均约 **342ms**、**P99 约 875ms**，说明在 **高并发短窗口** 下已有 **排队/尾延迟**，面试可主动说「未做生产调优、单机同机」。  
3. **与 4.3 对比**：服务层直连约 **1672 QPS**（500 并发、299ms）；HTTP 层 **3000 样本** 下 **~1910/s** 为另一套负载模型，**不可简单数值对齐**，统一说成「本地多轮基线」即可。

#### 4.4.4 面试一句话（HTTP 压测）

`我在本地用 JMeter 配了多 token CSV，对 POST 秒杀接口做了 HTTP 层压测，响应断言里把受理成功、不能重复下单、库存不足都算作有效响应，避免把业务规则当成失败。在一轮 3000 次请求的聚合报告里，吞吐大约一千九每秒、断言失败率为 0，P99 在几百毫秒量级；环境是单机本机，结果用于相对对比和面试说明，不作为生产 SLA。`

## 5. 当前测试结果的整体分析

### 5.1 当前结果说明了什么

综合当前三轮测试，可以得出几个结论：

#### 1. ID 生成链路没有成为瓶颈

- `RedisIdWorker` 3 万次请求约 1.386 秒完成
- 吞吐约 2.16 万 QPS
- 远高于当前秒杀服务层基线 QPS

#### 2. 单用户一人一单逻辑有效

- 500 并发下 1 成功、499 重复下单
- 说明重复请求拦截逻辑是生效的

#### 3. 多用户并发秒杀入口逻辑有效

- 500 不同用户并发下全部成功
- 说明 Redis Lua 校验链路通畅

#### 4. 当前测试阶段没有出现明显异常

- `error = 0`
- 没有线程卡死
- 没有任务遗漏
- 应用在测试结束后正常关闭

### 5.2 当前结果不能说明什么

这点非常重要，避免误判。

当前这些测试 **不能直接说明**：

1. Kafka Consumer 最终落库吞吐上限
2. 端到端 HTTP 接口「生产级」QPS 上限（**已做本地 JMeter 基线**：见 **4.4**，仍为**单机相对值**）
3. Kafka 消息堆积时的消费恢复能力
4. 数据库在持续高压下的稳定性
5. 手动 ack、死信、补偿机制的效果

因为早期测试主要是：

- 直接在服务层调用 `voucherOrderService.seckillVoucher(...)`
- 更偏向验证入口链路

后续已补充 **4.4 HTTP 层 JMeter 压测**；**端到端落库全链路极限**仍需结合 Kafka lag、DB 监控单独评估。

## 6. 当前测试结论

### 结论一：当前秒杀入口已经具备可用性

从当前结果来看：

- Redis 库存预热正常
- Redis Lua 秒杀入口正常
- 单用户重复请求能被拦截
- 多用户并发请求能正常受理

说明：

- 当前秒杀入口链路已经具备可用性

### 结论二：当前服务层秒杀入口吞吐基线大致在千级 QPS；HTTP 层 JMeter 另有单轮 ~1.9k/s 量级

当前测试结果显示：

- 单用户并发场景：约 2427 QPS
- 多用户并发场景：约 1672 QPS
- **HTTP 层 JMeter（4.4 场景 D）**：3000 样本下聚合 **Throughput ~1910/s**（单机，0% 断言失败）

这说明：

- 当前服务层秒杀入口的基础处理能力已经达到千级 QPS；**真实 HTTP** 在本地一轮高并发下亦出现 **~1.9k/s** 量级（与机器、Ramp-up、断言配置强相关）

但这个结果仍然只是：

- 当前机器
- 当前本地环境
- 当前测试方式

下的基线值，不应直接视为生产吞吐上限。

### 结论三：后续优化重点应该转向链路可靠性和端到端压测

既然当前入口已经能跑通，接下来的重点就不应该只盯着“能不能秒杀成功”，而应该继续推进：

1. Kafka Consumer 改成手动 ack
2. 增加 traceId 和成功/失败埋点
3. 增加数据库唯一索引 `unique(user_id, voucher_id)`
4. 增加 Kafka 重试、死信和补偿机制
5. 做 HTTP 层压测和端到端落库压测
6. 观察 MySQL、Redis、Kafka 在持续压力下的瓶颈位置
