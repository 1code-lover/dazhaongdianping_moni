# hm-dianping 限流设计方案
## 1. 文档目的

本文档用于指导 `hm-dianping` 项目第一版限流能力落地，目标是在**不大幅改造现有业务链路**的前提下，为高频接口、登录验证码接口和秒杀接口补齐一套可实施、可解释、可逐步演进的限流方案。

这份文档重点回答 5 个问题：

- 为什么这个项目现在需要限流
- 限流应该放在哪几层
- 哪些接口先做，规则怎么定
- 代码结构怎么设计
- 后续如何从轻量方案升级到更成熟的治理方案

本文档关注的是**第一版工程落地方案**，不是 Sentinel 控制台接入方案，也不是完整的服务治理平台方案。

## 2. 当前项目现状

结合当前代码，项目已经具备一定的高并发保护能力，但还没有形成完整的限流体系。

当前已有的能力：

- `VoucherOrderServiceImpl.seckillVoucher(...)` 通过 Redis Lua 脚本完成秒杀资格校验
- `seckill.lua` 已经实现库存校验、一人一单校验、库存预扣和异步投递
- Kafka 链路已经用于秒杀订单异步削峰
- Web 层已接入登录拦截器和 token 刷新拦截器
- Nginx 已经承担前后端静态资源和 `/api` 反向代理

当前缺失的能力：

- 没有通用接口限流拦截器
- 没有按 IP、用户、手机号等维度的访问频率控制
- 没有验证码发送冷却时间和窗口次数限制
- 秒杀入口有库存和重复下单保护，但没有入口请求频率限制
- Nginx 层没有 `limit_req` 或 `limit_conn` 配置
- 限流失败没有统一错误码和统一日志埋点

一句话概括当前状态：

`项目有并发保护，但没有完整限流体系。`

## 3. 方案目标

第一版限流方案只追求三件事：

1. 保护核心接口，避免异常流量直接打穿应用
2. 对高风险接口补齐访问频率约束，降低刷接口和误用风险
3. 保持设计简单，便于快速落地、联调和压测验证

落地后希望达到的效果：

- `/user/code` 不再允许被短时间重复刷取验证码
- `/user/login` 具备基础防爆破能力
- 秒杀接口在进入 Lua 资格校验前先做入口限流
- 高频写接口具备基础防刷能力
- 限流命中时可以统一返回、统一记录、统一调参

## 4. 设计原则

- 分层治理：Nginx 做粗粒度限流，应用层做细粒度限流
- 业务优先：按接口特性定规则，而不是全站统一阈值
- 轻量优先：第一版使用 `Spring Boot + Redis` 落地，不引入重型治理平台
- 低侵入：尽量通过注解、拦截器、AOP 方式接入，不改动大量业务代码
- 易演进：保留未来迁移到 Sentinel 或网关治理的空间

## 5. 总体方案

第一版采用“三层限流”设计：

1. Nginx 层：做粗粒度入口保护
2. Spring Boot 层：做业务感知的细粒度限流
3. 秒杀业务层：保留现有 Lua 资格校验与异步削峰，并新增秒杀入口限流

整体链路如下：

```text
客户端请求
-> Nginx 粗粒度限流
-> Spring MVC / AOP 细粒度限流
-> Controller
-> Service
-> 秒杀场景下继续进入 Redis Lua + Kafka 异步链路
```

这三层的职责划分如下：

- Nginx：挡住明显异常流量，减少无效请求进入 JVM
- Spring Boot：按用户、手机号、接口、活动维度做业务限流
- 秒杀业务层：继续负责资格校验、库存预扣、防重复和削峰

## 6. 为什么第一版不直接上 Sentinel

Sentinel 是可选升级方向，但不建议作为第一版直接接入，原因如下：

- 当前项目里还没有现成的治理中台和动态规则管理能力
- 现在最急的是尽快补齐限流能力，而不是先搭复杂平台
- Redis 版限流更贴近现有架构，改造成本低，调试成本也低
- 现阶段的限流需求主要是验证码、登录、防刷和秒杀入口保护，用轻量方案足够覆盖

因此第一版建议先定为：

`Nginx + Spring 注解/AOP + Redis`

后续如果需要控制台动态调参、熔断降级、热点参数治理，再考虑升级到 Sentinel。

## 7. 接口分级与限流策略

### 7.1 登录与验证码类接口

典型接口：

- `/user/code`
- `/user/login`

这类接口最容易被脚本恶意刷取，需要优先治理。

建议规则：

| 接口 | 维度 | 规则建议 | 说明 |
|-----|-----|-----|-----|
| `/user/code` | 手机号 | 30 秒内 1 次 | 发送冷却时间 |
| `/user/code` | 手机号 | 60 秒内最多 3 次 | 防止重复刷短信 |
| `/user/code` | IP | 60 秒内最多 10 次 | 防止单 IP 横向刷号 |
| `/user/login` | 手机号 | 60 秒内最多 20 次 | 防止撞库和暴力尝试 |
| `/user/login` | IP | 60 秒内最多 30 次 | 防止单 IP 高频探测 |

### 7.2 普通读接口

典型接口：

- `/shop/**`
- `/shop-type/**`
- `/blog/hot`

这类接口以保护系统稳定性为主，不宜限得过严。

建议规则：

- 按 IP 限流
- 默认 `1 秒内 50~100 次`
- Nginx 层允许一定突发 `burst`

### 7.3 普通写接口

典型接口：

- 点赞
- 评论
- 发帖
- 上传
- 关注
- 签到

这类接口既要保护系统，也要防刷行为。

建议规则：

| 接口类别 | 维度 | 规则建议 |
|-----|-----|-----|
| 点赞/取消点赞 | 用户 ID | 1 秒内最多 10 次 |
| 评论 | 用户 ID | 1 分钟内最多 20 次 |
| 发帖 | 用户 ID | 1 分钟内最多 5 次 |
| 上传 | 用户 ID | 1 分钟内最多 10 次 |
| 关注/取关 | 用户 ID | 1 秒内最多 5 次 |
| 签到 | 用户 ID | 1 分钟内最多 5 次 |

未登录场景下可以退化为按 IP 限流。

### 7.4 秒杀接口

典型接口：

- `/voucher-order/seckill/**`

当前秒杀链路已经有后置保护，但缺少前置入口限流。

建议补充规则：

| 维度 | 规则建议 | 说明 |
|-----|-----|-----|
| 用户 ID | 1 秒内最多 1~3 次 | 防止同一用户短时间狂点 |
| IP | 1 秒内最多 20~50 次 | 防止单 IP 请求洪峰 |
| `voucherId + userId` | 1 秒内最多 1 次 | 进一步约束热点券重复点击 |
| `voucherId` | 可配热点保护阈值 | 活动维度保护 |

注意：

- 秒杀限流并不替代现有 Lua 校验
- 限流只负责拦入口流量
- Lua 仍然负责库存、一人一单和异步投递

## 8. 第一版范围

### 8.1 本次优先落地的接口

- `/user/code`
- `/user/login`
- `/voucher-order/seckill/**`

### 8.2 第二批建议接入的接口

- 点赞相关接口
- 评论相关接口
- 发帖相关接口
- 上传接口

### 8.3 第一版暂不处理的内容

- 后台管理接口单独治理
- 动态规则推送控制台
- 复杂滑动窗口或令牌桶集群治理
- 全链路降级熔断策略

原因很简单：

第一版先聚焦最值钱、风险最高、最容易验证收益的接口。

## 9. 技术选型

第一版技术选型如下：

- Web 接入方式：`Spring MVC 拦截器` 或 `AOP + 注解`
- 计数存储：`Redis`
- 统一异常：项目现有 `Result` 结构 + 全局异常处理
- 入口粗限流：`Nginx limit_req`

在应用层，我更推荐：

`自定义注解 + AOP`

原因：

- 能做到规则就近声明
- 接口可读性更好
- 复用性更高
- 后续更容易扩展多种维度和多种限流策略

## 10. 限流算法选择

### 10.1 第一版算法

第一版优先使用：

- 固定窗口计数器
- 冷却时间控制

具体说明：

1. 固定窗口计数器

- 使用 Redis `INCR` 计数
- 首次访问时设置 `EXPIRE`
- 超过阈值则拒绝请求

优点：

- 实现简单
- 性能高
- 足够满足当前项目阶段需求

2. 冷却时间控制

适用于验证码发送这类“间隔限制”场景。

- 使用 Redis `SETNX` 或 Lua 实现
- 设置 30 秒冷却 key
- 冷却期内重复请求直接拒绝

### 10.2 为什么第一版不直接上滑动窗口或令牌桶

- 固定窗口已经足够满足当前阶段的主要场景
- 项目当前最需要的是先建立限流能力，不是追求算法最复杂
- 令牌桶更适合后续更精细的全站治理或网关统一治理

如果后续压测发现固定窗口边界效应明显，再局部升级为滑动窗口或 Lua 令牌桶即可。

## 11. 应用层设计方案

### 11.1 注解设计

建议新增 `@RateLimit` 注解，作用于 Controller 方法。

建议字段：

| 字段 | 类型 | 说明 |
|-----|-----|-----|
| `keyPrefix` | `String` | Redis key 前缀 |
| `timeWindowSeconds` | `int` | 时间窗口 |
| `maxRequests` | `int` | 窗口内允许请求数 |
| `limitType` | `LimitType` | 限流维度 |
| `message` | `String` | 命中后提示语 |
| `cooldownSeconds` | `int` | 冷却时间，可选 |
| `pathKey` | `boolean` | 是否拼接接口路径 |

建议支持的 `LimitType`：

- `IP`
- `USER`
- `PHONE`
- `IP_AND_PATH`
- `USER_AND_PATH`
- `CUSTOM`

### 11.2 核心组件拆分建议

建议新增以下组件：

#### `RateLimit`

- 自定义注解
- 用于在接口上声明限流规则

#### `LimitType`

- 枚举类
- 定义限流维度

#### `RateLimitAspect`

- AOP 切面
- 解析注解
- 生成限流 key
- 调用 Redis 限流逻辑
- 超限时抛出业务异常

#### `RateLimitKeyResolver`

- 负责根据上下文拼装限流 key
- 支持按 IP、用户、手机号、接口路径等维度生成 key

#### `RateLimitService`

- 对外提供限流判断能力
- 负责执行固定窗口计数和冷却判断

#### `RateLimitException`

- 限流命中时抛出的业务异常

#### `WebExceptionAdvice`

- 统一处理 `RateLimitException`
- 返回统一错误结构

### 11.3 接入层建议

第一版建议只在 Controller 层接注解，业务代码最少改动。

示例思路：

```java
@PostMapping("/code")
@RateLimit(keyPrefix = "user:code:phone", limitType = LimitType.PHONE, timeWindowSeconds = 60, maxRequests = 3, cooldownSeconds = 30, message = "验证码发送过于频繁")
public Result sendCode(...) {
    return userService.sendCode(...);
}
```

秒杀接口同理：

```java
@PostMapping("/seckill/{id}")
@RateLimit(keyPrefix = "seckill:user", limitType = LimitType.USER_AND_PATH, timeWindowSeconds = 1, maxRequests = 3, message = "秒杀请求过于频繁")
public Result seckillVoucher(...) {
    ...
}
```

## 12. Redis Key 设计

建议统一命名风格：

`rl:业务:维度:标识`

典型示例：

- `rl:user:code:phone:13800138000`
- `rl:user:code:ip:192.168.1.10`
- `rl:user:login:phone:13800138000`
- `rl:user:login:ip:192.168.1.10`
- `rl:seckill:user:10086`
- `rl:seckill:ip:192.168.1.10`
- `rl:seckill:voucher:12:user:10086`

冷却时间 key 示例：

- `rl:cooldown:user:code:phone:13800138000`

设计要求：

- key 前缀统一，便于排查和监控
- key 能直观看出业务含义
- 不把过多动态参数混进 key，避免粒度失控

## 13. Redis 执行逻辑建议

### 13.1 固定窗口计数器

逻辑如下：

1. 对目标 key 执行 `INCR`
2. 如果返回值为 `1`，则设置 `EXPIRE`
3. 如果计数值大于阈值，则判定超限

这个方案简单直接，适合第一版快速落地。

### 13.2 冷却时间逻辑

逻辑如下：

1. 对冷却 key 执行 `SETNX`
2. 如果设置成功，则补 `EXPIRE`
3. 如果设置失败，则说明仍在冷却期，直接拒绝

### 13.3 是否使用 Lua

建议：

- 第一版固定窗口可以先用简单命令实现
- 如果后续想把 `INCR + EXPIRE` 和冷却控制做成严格原子操作，可以收敛到一个 Lua 脚本

换句话说：

第一版不强制要求单独写限流 Lua，但架构上要保留这个升级位。

## 14. Nginx 层方案

Nginx 不做复杂业务限流，只做入口保护。

建议两类规则：

1. `/api` 总入口限流
2. 高风险接口专项限流

建议方向：

- `/api/`：按 IP 做基础限流
- `/api/user/code`：更严格的 IP 限流
- `/api/voucher-order/**`：秒杀入口单独限流

Nginx 的角色是：

- 尽量在请求进入 JVM 前过滤明显异常流量
- 降低线程池、连接数和应用资源浪费

但不要依赖 Nginx 去做：

- 按用户 ID 限流
- 按手机号限流
- 按 `voucherId` 热点限流

这些业务规则应由应用层处理。

## 15. 统一返回与异常处理

建议限流命中后统一返回“请求过于频繁”语义。

建议统一内容：

- 业务码：沿用项目 `Result.fail(...)` 结构
- HTTP 语义：建议保持 200 + 业务失败，或者后续逐步升级为 429
- 业务文案：接口可自定义，默认统一为“请求过于频繁，请稍后再试”

建议同时在全局异常处理中统一兜底：

- 拦截 `RateLimitException`
- 统一打日志
- 统一返回结果

## 16. 日志与监控建议

第一版即使不接 Prometheus，也建议把限流日志打全。

每次命中限流建议记录：

- 接口路径
- 限流规则名
- 用户 ID
- IP
- 手机号
- Redis key
- 当前计数
- 时间窗口
- 阈值

日志价值：

- 压测时看是否误伤正常流量
- 联调时定位规则是否生效
- 上线后辅助调参

后续如需扩展监控，可增加：

- 限流命中次数
- 各接口命中分布
- 秒杀接口被拦截比例

## 17. 建议实施顺序

建议按下面顺序推进：

### 第一阶段：基础框架落地

- 新增限流注解
- 新增限流切面
- 新增 Redis 限流服务
- 新增统一异常类和统一返回
- 完成通用限流 key 解析能力

### 第二阶段：高风险接口接入

- 接入 `/user/code`
- 接入 `/user/login`
- 接入 `/voucher-order/seckill/**`

### 第三阶段：高频写接口接入

- 点赞
- 评论
- 发帖
- 上传

### 第四阶段：Nginx 粗限流补齐

- 增加 `/api` 总入口限流
- 增加 `/user/code` 和秒杀接口专项限流

### 第五阶段：压测与调参

- 验证规则是否生效
- 看是否误伤正常请求
- 调整窗口和阈值

## 18. 联调与验证建议

落地后建议至少验证以下内容：

1. `/user/code` 在 30 秒内重复发送时是否被拒绝
2. `/user/code` 在 60 秒内超过 3 次时是否被拒绝
3. `/user/login` 高频请求是否被限制
4. 秒杀接口在高频点击下是否先被入口限流拦截
5. 未命中限流时原有业务逻辑是否不受影响
6. 限流异常是否统一返回
7. 日志中是否能清晰看到命中规则和维度

## 19. 风险与取舍

第一版方案的主要取舍如下：

### 19.1 优点

- 改造成本低
- 与现有 Redis 架构贴合
- 实现速度快
- 足够覆盖当前阶段主要问题

### 19.2 局限

- 固定窗口存在边界效应
- 规则变更需要改代码或改配置，不如治理平台灵活
- 复杂热点参数治理能力有限
- 还没有完整监控大盘和动态控制台

这不是问题，而是第一版刻意控制复杂度后的结果。

## 20. 后续演进方向

如果第一版跑通，后续可以按下面方向演进：

1. 固定窗口升级为滑动窗口或 Lua 令牌桶
2. 限流规则外置到配置文件
3. 支持更丰富的自定义 key 维度
4. 接入监控指标和告警
5. 视项目复杂度升级到 Sentinel

推荐演进路径：

`Redis 注解限流 -> Redis Lua 精细限流 -> Sentinel 动态治理`

## 21. 最终建议

对于当前 `hm-dianping` 项目，第一版最合适的限流落地方式不是“大而全”的治理平台，而是：

`Nginx 粗限流 + Spring 注解限流 + Redis 计数控制`

最推荐的首批落地点：

- `/user/code`
- `/user/login`
- `/voucher-order/seckill/**`

这样做的优点是：

- 能快速补齐明显短板
- 不破坏现有秒杀主链路
- 方案简单、工程可落地、便于讲清楚

后续如果你认可这份设计，我们就按这份文档继续往下做实现。
