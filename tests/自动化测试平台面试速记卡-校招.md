# 自动化测试平台面试速记卡（校招）

## 30秒版（电梯自我介绍）

我在 `hm-dianping` 项目里做了一套 API+UI 一体化自动化测试平台。  
技术上用 `PyTest + Selenium`，并用 `Page Object + fixture` 解耦页面元素、业务动作和用例断言。  
我还做了自动鉴权：发验证码、从 Redis 读验证码、自动登录拿 token。  
目前全量稳定结果是 `10 passed, 1 skipped`，并且支持一键执行和 Allure 报告。


## 60秒版（标准回答）

这个项目秒杀链路比较复杂，涉及 Redis、Kafka、死信补偿。  
我搭平台时重点做了三件事：

1. **框架分层**：PyTest + Selenium，Page Object 把定位和断言解耦，后续维护成本低。  
2. **自动鉴权**：调用 `/user/code` 发码，从 Redis 读 `login:code:{phone}`，再调 `/user/login` 拿 token，API 和 UI 共用。  
3. **工程化落地**：加一键脚本做端口预检和全量执行，结果统一输出到 Allure。  

产出上，已经覆盖登录鉴权、未登录拦截、一人一单、登出失效、UI 秒杀主流程等高价值场景，回归可重复执行。


## 3分钟版（建议主讲）

我在这个项目里做的不是“写几条测试脚本”，而是搭了一个能长期演进的测试底座。  
背景是秒杀业务链路复杂，单靠手工回归效率低、漏测风险高。

我先做了技术方案：

- PyTest 做统一执行入口
- Selenium 做 UI 自动化
- Page Object + fixture 做三层解耦
- Allure 统一报告，失败重试提升稳定性

然后我做了项目特定能力：

- 自动鉴权链路：`/user/code` -> Redis 取验证码 -> `/user/login` 拿 token
- token 同时给 API 和 UI 用例复用
- 自动发现可用秒杀券并校验活动时窗

为了让这套东西真正“能用”，我加了一键脚本：

- 先做 8080/8081/6379（可选 8082）端口预检
- 再按模式执行 API/UI/全量
- 自动产出 Allure 原始结果目录

最终稳定基线是 `10 passed, 1 skipped`，覆盖了核心链路：

- 鉴权正反向
- 秒杀一人一单
- token 生命周期（登录/登出）
- UI 未登录拦截与已登录下单反馈

后续扩展计划是先到 30+ 条，优先补边界和异常路径，再接 CI 做持续回归。


## 高频追问一句话

- **为什么只有10条？**  
  先做高价值稳定基线，再规模扩展，这是工程上更可持续的路线。

- **死信是不是系统挂了？**  
  不一定，像 `ROLLBACK_DONE + redisRollback=OK` 说明补偿闭环完成，重点看是否高频。

- **你的亮点是什么？**  
  把手工验证变成可复用的工程化回归能力：自动鉴权 + 一键执行 + 报告统一。
