# HM-DianPing 自动化测试平台实现方案（面试版）

## 1. 项目背景与目标

本项目是典型的高并发秒杀业务，技术栈包含 Spring Boot、Redis、Kafka、Nginx，前端为静态页面 + Axios 调用后端接口。  
业务链路中最关键的是：

- 登录鉴权（验证码 + token）
- 秒杀预扣与异步下单
- 死信补偿与 Redis 回滚

自动化测试平台目标不是“把所有用例一次性写完”，而是先构建一套可持续演进的测试底座：

- 框架标准化：PyTest + Selenium + Page Object + fixture
- 稳定性：失败重试、环境预检、自动鉴权
- 可观测：Allure 统一报告（API + UI）
- 可执行：一键脚本跑全链路


## 2. 测试平台总体设计

### 2.1 技术选型

- 测试框架：`pytest`
- UI 自动化：`selenium`
- 页面模型：`Page Object`
- HTTP 测试：`requests`
- 报告：`allure-pytest`
- 重试：`pytest-rerunfailures`
- 鉴权辅助：`redis`（自动读取验证码）

### 2.2 分层结构（解耦思路）

三层解耦如下：

- 元素层（Page）
  - 只维护页面元素定位和原子操作
  - 目录：`tests/ui/pages/`
- 业务动作层（Page 方法）
  - 把“登录”“点击秒杀”封装成业务动作
  - 仍放在 Page 类，避免用例直接操作元素
- 用例层（Case）
  - 只关注业务断言，不关心 DOM 细节
  - 目录：`tests/ui/cases/`、`tests/api/`

### 2.3 运行拓扑（与项目实际一致）

- 前端：`127.0.0.1:8080`（Nginx）
- 后端：`127.0.0.1:8081`、`127.0.0.1:8082`
- Redis：`127.0.0.1:6379`
- Nginx `/api` 反向代理后端双节点

这也是为什么测试脚本加入了端口检查和 `-Require8082` 选项：避免 upstream 半可用造成随机失败。


## 3. 代码级实现细节

## 3.1 配置与依赖

- `requirements-test.txt`
  - 固化测试依赖，降低环境差异
- `pytest.ini`
  - 统一 marker（`api`、`ui`）
  - 默认重试（`--reruns 1`）

## 3.2 fixture 设计（核心）

文件：`tests/conftest.py`

关键 fixture：

- `api_session`：复用 HTTP 会话
- `driver`：Selenium 浏览器实例
- `auth_token`：自动获取 token（可被手工 token 覆盖）
- `token_factory`：可生成多个独立用户 token，解决“一人一单”互相污染
- `seckill_voucher`：自动扫描可用秒杀券
- `active_seckill_voucher`：按 begin/end 时间过滤活动有效券

关键设计点：

- **自动鉴权链路**
  - 调 `/user/code` 发码
  - 从 Redis 读取 `login:code:{phone}`
  - 调 `/user/login` 获取 token
- **驱动稳定性回退**
  - 先走 `webdriver-manager`
  - 失败则尝试本地 `chromedriver`
  - 再失败走 Selenium 默认发现
  - 仍失败则 skip（避免整套因外网驱动下载失败崩掉）
- **失败证据采集**
  - UI 失败自动附加截图与 page source 到 Allure

## 3.3 Page Object 实现

文件：

- `tests/ui/pages/base_page.py`
- `tests/ui/pages/login_page.py`
- `tests/ui/pages/shop_detail_page.py`

关键实践：

- 基类统一封装 `open/click/type/text_of`
- 登录页处理 ElementUI 单选按钮在 headless 下“可见但不可点击”的问题，增加 JS 兜底
- 秒杀页增加“按钮就绪等待”，按钮不可用时输出可读性 skip 原因

## 3.4 API 用例实现

文件：`tests/api/test_hmdp_api_smoke.py`

覆盖重点：

- `/shop-type/list` 基础可用性与数据非空
- `/user/me` 未登录 401
- 自动登录后 `/user/me` 成功
- 自动发现秒杀券并验证券列表可见
- 同用户重复秒杀被拦截
- 登出后 token 失效
- 非法 token 秒杀被拒绝

## 3.5 UI 用例实现

文件：

- `tests/ui/cases/test_login_page_smoke.py`
- `tests/ui/cases/test_seckill_flow_ui.py`

覆盖重点：

- 登录页基础可访问性
- 未登录点击秒杀跳登录页
- 已登录点击秒杀出现业务反馈（成功/重复/库存不足）

说明：

- `test_login_action_template` 默认关闭（需 `RUN_UI_LOGIN_E2E=1`），防止环境不稳定时影响主回归。

## 3.6 一键执行脚本

文件：`scripts/run-tests.ps1`

功能：

- 启动前端/后端前的端口预检（8080/8081/6379）
- 可选强校验 8082（`-Require8082`）
- 支持模式：
  - `all`
  - `api`
  - `ui`
  - `seckill`
- 自动生成 `reports/allure-results`
- 可选直接 `allure serve`


## 4. 当前“10条通过”的解释（面试可用）

最近稳定回归结果是：

- `10 passed, 1 skipped`

这不是“只会写 10 条”，而是刻意分阶段：

- 第一阶段先做 **高价值冒烟集（Smoke Baseline）**
- 目标是先把链路跑通、结果稳定、可持续执行
- 在此基础上再扩用例数量，比一开始堆很多脆弱用例更工程化

当前 10 条覆盖已经包含：

- 登录鉴权正反向
- token 生命周期（登录/登出）
- 秒杀核心约束（一人一单）
- UI 关键入口（未登录拦截、已登录下单反馈）
- 业务可用性（券查询、类目查询）


## 5. 为什么会有 `tb_voucher_order_fail_task`

这张表是本项目“异步失败补偿”的关键治理能力，不是异常本身：

- 当 Kafka 消费端多次重试后仍失败
- 消息进入 DLT（死信队列）
- 系统把失败记录落入 `tb_voucher_order_fail_task`
- 同时触发 Redis 回滚（库存 + 一人一单标记）

`status=ROLLBACK_DONE` + `failure_type=NON_RECOVERABLE` 的含义：

- 本条消息主链路最终失败
- 但补偿已经执行完成
- 数据一致性已尽量对齐

这在高并发系统中是“可观测、可追踪、可补偿”的正确设计，不是简单报错。


## 6. 扩展方案（从 10 条到 30+）

建议按优先级迭代，而不是一次铺太多：

### P0（先补 8~10 条）

- 秒杀活动未开始/已结束/库存为 0 的边界断言
- 同券不同用户并发抢购结果分布校验
- DLT 落库字段完整性校验（traceId/orderId/errorMsg）
- 手动重试任务接口与状态流转校验

### P1（再补 8~12 条）

- 店铺列表分页/排序（距离、人气、评分）
- 详情页券展示与库存变更可见性
- 登录页异常手机号、错误验证码场景
- token 过期刷新链路（若接入网关/拦截器续期策略）

### P2（持续建设）

- CI 定时回归（白天增量 + 夜间全量）
- 压测后冒烟（loadtest + regression）
- Allure 历史趋势看板

## 8. 常见追问与回答思路

### Q1：为什么不是一开始就 50 条？

答：先做稳定基线。测试平台是工程，不是脚本堆叠。先把 10 条高价值链路跑稳，再规模化扩展，维护成本最低。

### Q2：你怎么处理 UI 用例脆弱问题？

答：Page Object 隔离定位变化；关键动作加等待与兜底；环境不满足时 skip 并给出原因，不让假红影响回归判断。

### Q3：为什么要接 Redis？

答：这个系统登录验证码存 Redis，接入 Redis 后可自动鉴权，避免人工介入，使回归可无人值守。

### Q4：死信日志算失败吗？

答：看状态。`ROLLBACK_DONE` 说明主链路失败但补偿完成；是系统韧性机制的一部分。关键是监控频率与根因分类。


## 9. 执行命令（面试演示可用）

```bash
# 全量（推荐）
powershell -ExecutionPolicy Bypass -File scripts/run-tests.ps1 -Mode all -Require8082

# 秒杀专项
powershell -ExecutionPolicy Bypass -File scripts/run-tests.ps1 -Mode seckill -Require8082

# UI + 开启真实登录动作
powershell -ExecutionPolicy Bypass -File scripts/run-tests.ps1 -Mode ui -EnableUiLoginE2E

# 查看报告
allure serve reports/allure-results
```


## 10. 结论

这套平台已经具备“可落地、可复现、可扩展、可面试展示”的四个特征：

- 可落地：和当前项目真实链路打通（Nginx + Redis + Kafka + 秒杀）
- 可复现：一键脚本、端口预检、统一依赖
- 可扩展：分层清晰，新增用例成本低
- 可展示：Allure 报告 + 死信补偿可观测，能讲清工程价值
