# 自动化测试平台面试讲稿

## 1) 3分钟口述版（先讲这个）

我在 `hm-dianping` 这个项目里，搭建了一套 API + UI 一体化自动化测试平台，目标是把秒杀核心链路做成可回归、可观测、可扩展。

技术上我选了 `PyTest + Selenium`，并按 `Page Object + fixture` 做分层：  
页面元素和业务动作放在 Page 层，用例层只写业务断言，避免 DOM 变动直接影响用例。

这个项目的难点是登录依赖验证码，我做了自动鉴权链路：  
先调用 `/user/code` 发码，再从 Redis 读取 `login:code:{phone}`，然后调用 `/user/login` 自动拿 token。  
这样 API 和 UI 用例都不需要人工输入验证码，回归可以无人值守执行。

我还把稳定性做了两块增强：  
第一是失败重试（`pytest-rerunfailures`），第二是一键执行脚本会先做端口预检，避免 Nginx upstream 半可用导致随机失败。  
报告统一用 Allure，API 和 UI 结果合并到一个报告目录。

当前稳定基线是 `10 passed, 1 skipped`，覆盖了登录鉴权、未登录拦截、一人一单、登出失效、秒杀 UI 主流程等高价值路径。  
不是只会 10 条，而是先做稳定冒烟基线，再按优先级扩到 30+ 条，这样工程上更可持续。


## 2) 10分钟深入版（面试官追问时展开）

### 2.1 需求拆解

我把需求拆成四块：

- 测什么：核心业务链路，而不是纯页面点点点
- 怎么测：API + UI 协同，不做单腿走路
- 怎么稳：重试、环境预检、自动鉴权、失败附件
- 怎么用：一键脚本 + 报告输出，团队可复用

### 2.2 架构与目录

- `tests/api/`：接口用例
- `tests/ui/pages/`：页面对象
- `tests/ui/cases/`：UI 用例
- `tests/conftest.py`：全局 fixture
- `scripts/run-tests.ps1`：一键执行

核心思想是三层解耦：元素、动作、断言。

### 2.3 自动鉴权关键设计

这个项目登录是短信验证码模式，传统 UI 自动化依赖人工输入，不稳定且低效。  
我改为：

1. `/user/code?phone=...` 发码
2. Redis 读取 `login:code:{phone}`
3. `/user/login` 获取 token
4. token 注入 API header 或 UI `sessionStorage`

这样把“人手操作”变成“系统动作”，大幅提升可回归性。

### 2.4 稳定性治理

我主要处理了三类不稳定源：

- **环境不一致**：脚本先查 8080/8081/6379，可选强校验 8082
- **浏览器驱动问题**：driver 初始化做多级回退（在线下载 -> 本地 -> 默认发现）
- **UI时序波动**：关键动作加显式等待和兜底 skip，避免假红

并且失败自动附带截图和 page source，方便直接定位。

### 2.5 业务覆盖策略

先做高价值冒烟：

- `/user/me` 正反向鉴权
- 秒杀券自动发现
- 同用户重复秒杀拦截
- 登出后 token 失效
- UI 未登录秒杀跳登录
- UI 已登录秒杀业务反馈

这套先跑稳，再按优先级扩容。

### 2.6 与死信补偿联动

我在讲测试平台时会提到系统韧性：  
当 Kafka 消费失败进 DLT，`tb_voucher_order_fail_task` 会落失败任务并触发 Redis 回滚。  
像 `ROLLBACK_DONE`、`redisRollback=OK` 表示补偿完成，是可观测可治理，而不是单纯“报错”。

这能体现我不仅会写脚本，也能结合业务链路判断系统状态。

### 2.7 结果与价值

- 形成了“一键执行 + 报告输出”的团队化能力
- 回归从人工验证转为可重复执行
- 面对复杂链路（Redis/Kafka/秒杀）能提供可观测闭环
- 后续扩展成本低，新增用例只需补 Page 或断言


## 3) 高频追问与回答模板

### Q1：为什么不用 JMeter/只做接口压测？

压测和回归不是替代关系。  
JMeter解决容量问题，PyTest+Selenium解决功能正确性和回归效率。  
我这套平台是“功能基线”，可和压测串联使用。

### Q2：为什么现在只有 10 条？

这是有意做“稳定冒烟基线”。  
先把核心链路跑稳，解决环境和框架稳定性，再规模化扩用例；  
否则一开始堆很多不稳定脚本，维护成本很高，团队很快弃用。

### Q3：怎么保证不是假通过？

- API 用例覆盖正反向断言（401、重复下单、登出失效）
- UI 覆盖关键入口行为（跳转、业务提示）
- 自动鉴权走真实后端链路，不是mock token
- 失败有截图和页面源码，便于复盘

### Q4：`tb_voucher_order_fail_task` 出现记录是不是系统有问题？

不一定。  
关键看状态：`ROLLBACK_DONE` + `redisRollback=OK` 表示补偿闭环已完成。  
如果持续增长才需要根因治理（消费端异常分类、重试策略、下游稳定性）。

### Q5：如何扩展到 30+？

- P0：秒杀边界 + 失败任务重试链路
- P1：店铺列表分页/排序 + 登录异常输入
- P2：CI 定时回归 + Allure 历史趋势


## 4) 你可以直接照读的结尾

我做这个平台的核心不是“写了几条脚本”，而是把测试能力工程化：  
能自动鉴权、能统一报告、能一键执行、能结合死信补偿判断真实系统状态。  
当前 10 条是稳定基线，接下来会按业务优先级扩到 30+，并接入 CI 做持续质量监控。
