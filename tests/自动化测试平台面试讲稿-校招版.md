# 自动化测试平台面试讲稿（校招风格）

## 1. 60秒自我介绍版（开场）

在这个 `hm-dianping` 项目里，我做的是一个 API + UI 一体化自动化测试平台。  
我用 `PyTest + Selenium` 搭了框架，用 `Page Object + fixture` 做了解耦，把页面元素、业务动作、用例断言分开。  
为了让回归可无人值守，我实现了自动鉴权：调用发码接口、从 Redis 取验证码、自动登录拿 token。  
最后用 Allure 统一输出报告，并做了一键脚本用于端口预检和全量执行。  
目前稳定基线是 10 条高价值用例，后续按优先级扩展。


## 2. 3分钟主讲版（校招推荐）

这个项目的业务重点是秒杀链路，涉及 Redis 预扣、Kafka 异步消费、死信补偿。  
我做测试平台时，先明确目标不是“写很多脚本”，而是“搭一个能长期跑、团队可复用的测试底座”。

我主要做了四件事：

1. **框架搭建**  
   用 PyTest 做统一入口，UI 用 Selenium，按 Page Object 分层。  
   Page 层负责元素定位和动作，用例层只写业务断言，后续维护更轻。

2. **自动鉴权**  
   这个项目登录是验证码模式，我做了自动化流程：  
   `/user/code` 发码 -> Redis 读 `login:code:{phone}` -> `/user/login` 拿 token。  
   这样 API 和 UI 都能复用 token，不需要人工干预。

3. **稳定性治理**  
   - 用例失败重试  
   - UI 失败自动截图和保存页面源码  
   - 一键脚本先做端口检查，避免服务没起来就盲跑  
   - 对不稳定环境（如 upstream 节点未起）给出可读的 skip 原因

4. **结果可视化**  
   所有结果统一写入 `reports/allure-results`，可直接 `allure serve` 查看。  
   目前全量稳定结果是 `10 passed, 1 skipped`，覆盖登录、鉴权、秒杀核心约束和 UI 主路径。

我这次最大的收获是：  
不仅写了测试脚本，更把测试做成了“可执行、可复现、可扩展”的工程能力。


## 3. 面试官常问（校招简洁回答）

### Q1：为什么只做了10条？

我先做的是高价值稳定基线。  
先把核心链路跑稳，再扩展数量，比一开始堆很多不稳定脚本更工程化。

### Q2：你这套怎么体现业务理解？

我不只测页面是否可点，而是测秒杀关键约束：  
未登录拦截、一人一单、登出后 token 失效、异常链路补偿可观测。

### Q3：看到 `tb_voucher_order_fail_task` 是不是系统有问题？

不一定。  
如果是 `ROLLBACK_DONE` 且 `redisRollback=OK`，说明主链路失败后补偿闭环完成。  
要看是否高频出现，高频才需要专项治理。

### Q4：你个人在这个项目里最有价值的点是什么？

我把“手工验证”变成了“自动化回归能力”：  
自动鉴权 + 一键执行 + 报告输出 + 可扩展分层，这是团队可持续使用的。


## 4. 校招风格结尾（可直接背）

我在这个项目里不是单纯补了几条测试，而是搭了一个可以长期演进的自动化测试平台。  
现在已经能稳定覆盖核心链路，后续我会继续按优先级扩到 30+ 条，并接入 CI 做持续质量保障。
