# 秒杀并发与 Kafka 面试话术（1/3/8 分钟版）

## 1 分钟版（电梯陈述）

我这个项目的秒杀链路是 Redis + Kafka + MySQL 分层协作，不是简单把请求直接写数据库。  
入口阶段先用 Lua 在 Redis 里做原子校验，一次完成库存判断、一人一单判断和消息入流，成功后接口快速返回受理结果。  
后面通过 Redis Stream 中继到 Kafka，Kafka Consumer 异步落 MySQL，业务成功才手动 ack。  
为了可靠性，我加了有限重试和死信队列，重试超限会进入 DLT，再落失败任务表，支持自动重试和人工处理。  
所以这套方案的核心价值是：高并发下能扛流量，同时具备失败可恢复、可追踪、可治理能力。

## 3 分钟版（面试主答）

这套秒杀链路我主要做了四件事。

第一，入口抗并发。  
秒杀接口进来后，不直接查库写库，而是先执行 Redis Lua。这个脚本在 Redis 侧原子完成四步：检查库存、检查一人一单、扣减库存、写入 Stream。  
这一步把最容易成为瓶颈的并发判断前置到内存，接口只做“快速受理”，所以吞吐和响应都更稳。

第二，异步解耦。  
Lua 写入 `stream.orders` 后，由 Relay 组件把消息转发到 Kafka。这里我做了一个关键语义：只有 Kafka 发送成功才 ack Redis Stream；失败不 ack，消息会留在 pending-list，服务重启后还会补扫 pending-list。  
这样能降低中间链路丢消息的风险。

第三，消费可靠性。  
Kafka Consumer 采用手动 ack，只有订单真正处理成功才提交位点。失败时交给 `SeekToCurrentErrorHandler + FixedBackOff` 做有限次重试。  
如果还是失败，消息会进 DLT，不再阻塞主消费链路。

第四，失败治理闭环。  
DLT 消息会落失败任务表，并带 traceId、orderId、userId、voucherId 等关键信息。  
系统有定时任务自动重试失败任务，也提供人工接口做手动重试和人工忽略。  
另外我还补了 Kafka lag 巡检和 Redis/DB 库存对账，用于运维阶段及时发现异常。

总结就是：我不只是把 Kafka 接上，而是把秒杀做成了“可运行 + 可恢复 + 可治理”的完整链路。

## 8 分钟版（深度展开）

### 1. 为什么这样设计

秒杀场景有三个典型问题：高并发瞬时流量、重复下单、链路局部失败导致不一致。  
如果接口直接落库，在高峰下会被锁竞争和慢 SQL 拖垮。  
所以我把链路拆成“入口原子校验 + 异步落库 + 失败治理”三段。

### 2. 入口阶段：Redis Lua 原子校验

请求进入 `seckill/{id}` 后，服务层会先生成订单号，再执行 Lua。  
Lua 一次做完：

- 判断 `seckill:stock:{voucherId}` 是否有库存
- 判断 `seckill:order:{voucherId}` 是否已有该用户
- 扣减 Redis 库存，写入用户下单标记
- 把订单核心字段写入 `stream.orders`

这样“一人一单 + 扣库存 + 入队”不会被并发打断。  
接口成功语义是“受理成功”，不是“订单已落库成功”，这是典型异步架构语义。

### 3. 中继阶段：Redis Stream 到 Kafka

我没有让入口直接发 Kafka，而是在 Redis Stream 和 Kafka 之间加了 Relay：

- 消费组读 Stream
- 组装消息投递 Kafka
- 发送成功才 XACK
- 发送失败保留 pending-list，后续恢复

这层的价值是解耦和可恢复：即使 Kafka 短暂抖动，前端入口受理链路不一定立即崩溃，同时中继层能补偿未确认消息。

### 4. 消费阶段：手动 ack + 重试 + DLT

Consumer 用手动 ack，业务完成才确认。  
如果异常：

- 可恢复异常：不 ack，进入有限重试
- 不可恢复异常：先回滚 Redis 预扣，再 ack，避免毒消息无限重试
- 重试超限：进入 DLT（`seckill-order-dlt`）

这套策略避免了“消息已提交但订单没落库”的丢单风险，也避免了坏消息长期阻塞主分区。

### 5. 幂等与一致性保障

我做了多层防重：

- Redis 入口一人一单（Set）
- 消费侧业务判重（user_id + voucher_id）
- DB 唯一索引最终兜底

库存也做了双层保护：

- Redis 预扣用于高并发快速判断
- DB 条件扣减用于最终业务事实

当 DLT 或不可恢复异常出现时，会执行 Redis 回滚脚本，释放预扣库存和用户标记，减少长期脏状态。

### 6. 失败治理与运维能力

DLT 消息会落失败任务表，变成可管理工单。  
我提供了三类处理能力：

- 自动重试（定时扫描待重试任务）
- 人工重试
- 人工忽略并标记状态

另外做了两个运维维度：

- Kafka lag 巡检
- Redis/DB 秒杀库存对账

所以失败不是只打日志，而是能追踪、能重试、能干预。

### 7. 你可以主动说的改进项

如果面试官继续追问，我会说下一步优化：

- 重试策略从固定间隔升级为退避策略
- 异常分类更细化，精确区分可恢复/不可恢复
- 增加失败任务看板和告警阈值
- 对账结果驱动自动补偿流程

这能体现我不仅能把功能做出来，也能把系统往可运维方向推进。

## 高频追问一口答（补充）

### 问：为什么要手动 ack？

因为“拉到消息”不等于“业务成功”。手动 ack 可以把位点提交绑定在业务成功之后，避免提前提交导致丢单。

### 问：为什么需要 DLT？

为了隔离坏消息，防止主链路被持续失败消息拖住，同时保留失败现场，便于补偿和人工处理。

### 问：如何防止重复下单和重复消费？

入口 Redis Set 拦截、消费侧业务判重、数据库唯一索引兜底三层防重，保证至少一次投递语义下也不会生成重复订单。

## 模拟面试官追问脚本（20 问，追加版）

### 1) 问：你为什么不直接用 Kafka，非要先 Redis Stream？
答：入口先写 Stream 可以把受理链路和 Kafka 可用性解耦，Relay 层还能用 pending-list 做恢复，减少中间断点丢消息。  
反问点：贵司更偏“入口直发 MQ”还是“先本地/Redis 落盘再转发”？

### 2) 问：你接口返回成功，但订单可能没落库，这不是问题吗？
答：这是异步架构的受理语义。成功表示“抢购资格受理成功”，最终状态由异步链路完成，并通过失败任务和补偿保证一致性。  
反问点：你们当前接口语义是强一致返回还是受理返回？

### 3) 问：Lua 里做了哪些原子操作？
答：库存判断、一人一单判断、库存扣减、用户标记写入、订单消息写 Stream 都在同一脚本中完成。  
反问点：你们线上是否也把资格校验前置到 Redis 脚本？

### 4) 问：为什么消费者要手动 ack？
答：因为“拉到消息”不等于“业务成功”。手动 ack 把位点提交后移到落库成功之后，避免提前确认导致丢单。  
反问点：你们目前消费者是自动提交还是手动提交？

### 5) 问：重试策略是怎么做的？
答：主消费失败走 `SeekToCurrentErrorHandler + FixedBackOff` 有限重试，超限后进 DLT，避免坏消息长期阻塞主链路。  
反问点：你们是否区分可恢复异常和不可恢复异常？

### 6) 问：DLT 在你系统里的作用是什么？
答：DLT 不是丢弃区，而是失败隔离区。进入 DLT 后会落失败任务表，可自动重试也可人工处理。  
反问点：你们是否有死信治理看板？

### 7) 问：重复消费会不会导致重复订单？
答：有三层防重：Redis 一人一单、消费侧业务判重、DB 唯一索引兜底，至少一次投递下也不会生成重复订单。  
反问点：你们是更依赖业务判重还是数据库约束？

### 8) 问：为什么数据库还要再扣库存？Redis 不是已经扣了吗？
答：Redis 扣的是高并发预扣，DB 扣的是最终业务事实。两层职责不同，最终一致通过重试/回滚/对账保障。  
反问点：你们有没有做 Redis 与 DB 的定期对账？

### 9) 问：如果消息永远失败怎么办？
答：不会无限重试。超过阈值进入 DLT，落失败任务后状态转人工池，避免一条毒消息拖垮消费。  
反问点：你们对 poison message 的处置策略是什么？

### 10) 问：你怎么做回滚？
答：针对不可恢复异常和 DLT 场景，执行 Redis 回滚 Lua，释放库存和用户购买标记，减少不一致。  
反问点：你们回滚是同步做还是异步补偿做？

### 11) 问：为什么 Producer 要同步等待 send 结果？
答：为了保证“真正写入 Kafka”再确认成功，避免仅提交到客户端缓冲就误判成功。  
反问点：你们是否统计过发送超时和失败比例？

### 12) 问：Kafka key 为什么用 userId？
答：同用户消息更容易进入同分区，有助于降低该用户相关消息乱序影响。  
反问点：你们线上分区 key 主要按用户、订单还是店铺？

### 13) 问：自动重试和人工重试怎么配合？
答：先自动重试提升自愈率，超过阈值转 `MANUAL_HANDLE_REQUIRED`，再通过接口人工重试或人工忽略。  
反问点：你们是否有 SRE/运营介入流程？

### 14) 问：失败任务表存什么，价值在哪里？
答：存 traceId、订单信息、原始 payload、错误原因、状态、重试次数。价值是把失败从日志变成可治理工单。  
反问点：你们有没有统一失败任务中心？

### 15) 问：怎么验证这套链路真的有效？
答：通过 fail-once/fail-always 开关模拟失败，验证“失败重试、进 DLT、落失败任务、自动重试成功变 DONE”的完整路径。  
反问点：你们压测时会做故障注入吗？

### 16) 问：你做了哪些可观测能力？
答：Kafka lag 巡检、失败任务状态追踪、库存对账日志、traceId 贯穿日志。  
反问点：你们现网告警是基于 metrics 还是日志规则？

### 17) 问：这套方案最大的风险点是什么？
答：异步受理下，接口成功与最终成功存在时间差；所以必须有 DLT、补偿、对账和人工处理闭环。  
反问点：你们在业务上如何向前端表达“受理中”状态？

### 18) 问：如果让你再优化一步，你先做什么？
答：先做异常分级和退避重试（next_retry_time），减少无效重试和消息风暴。  
反问点：你们线上最痛的是重试风暴还是人工积压？

### 19) 问：为什么不用 RocketMQ？
答：不是绝对优劣，关键是现有生态和团队经验。当前基于 Spring Kafka 已满足吞吐、重试、死信、运维需求。  
反问点：贵司消息中间件选型更看重哪些维度？

### 20) 问：一句话总结你的亮点？
答：我把秒杀从“能跑”做到了“可恢复、可治理、可运维”，尤其是失败链路闭环和面向高并发的工程化设计。  
反问点：如果入职，我可以先帮你们把哪条链路的失败治理补齐？
