# S1 - Shop 查询基线压测报告

## 1. 测试目的

本次测试用于记录 `GET /shop/{id}` 在**当前版本**下的基线表现，为后续引入本地缓存后的对比压测提供参考依据。

说明：

- 当前版本**不是无缓存版本**
- 当前版本已具备 **Redis 单层缓存**
- 当前版本**尚未引入 L1 本地缓存**

因此，本次基线的准确含义是：

**Redis 单层缓存版本的基线压测结果**

## 2. 测试对象

- 接口：`GET /shop/{id}`
- 本次测试路径：`/shop/1`
- 服务地址：`http://localhost:8081`

## 3. 当前代码状态说明

结合当前项目代码，`shop` 查询接口已经接入 Redis 缓存：

- `ShopServiceImpl.queryById()` 当前调用 `CacheClient.queryWithPassThrough(...)`
- 查询链路为：Redis -> DB 回源
- 尚未接入 Caffeine 等本地缓存

因此，后续优化的目标不是“从无缓存变成有缓存”，而是：

**从 Redis 单层缓存，升级为 L1 本地缓存 + L2 Redis 二级缓存**

## 4. 测试过程记录

### 4.1 初始现象

在早期测试中，曾出现以下现象：

- 刚开始压测正常
- 运行一段时间后错误率显著上升
- JMeter 控制台出现：

`java.net.BindException: Address already in use: connect`

初步判断：

- 该问题更像是 **JMeter GUI 模式长时间压测导致的本地连接资源问题**
- 不能直接据此认定为业务代码问题

### 4.2 排查过程

为排除脚本配置或压测端问题，采用递进方式验证：

1. 使用小流量配置做稳定性验证
2. 逐步提升线程数与循环次数
3. 将关注点放在：
   - 是否持续报错
   - 是否仍出现 `BindException`
   - 错误率是否为 0

排查结论：

- 小流量验证通过
- 中等并发短时循环验证通过
- 当前 `GET /shop/1` 接口在现有代码下可稳定返回
- 早期高错误率更可能与压测执行方式相关，而不是接口本身先崩溃

## 5. 本轮有效结果

根据当前确认过的有效压测结果，可记录如下：

| 场景 | 接口 | 样本数 | 平均响应(ms) | 最大响应(ms) | 错误率 | 吞吐量 |
|---|---|---:|---:|---:|---:|---:|
| S1 Baseline | `GET /shop/1` | 10000 | 0~1 | 31 | 0.00% | 203.1/sec |

说明：

- 从截图结果看，本轮总样本数为 `10000`
- 平均响应时间接近 `0ms`
- 最大响应时间为 `31ms`
- 错误率 `0.00%`
- 吞吐量约 `203.1/sec`

该结果说明：

- 当前 Redis 单层缓存命中情况下，`shop` 详情查询性能较好
- 在当前测试配置下，接口稳定性正常

## 6. 阶段性结论

本轮基线压测可以得出以下结论：

1. `GET /shop/{id}` 当前并非无缓存，而是 Redis 单层缓存版本
2. 当前接口在有效测试条件下表现稳定，错误率可控制在 `0%`
3. 早期出现的大量错误，更可能来自 JMeter GUI 模式或本地连接资源限制
4. 当前结果可以作为后续“引入本地缓存后”的对照基线之一

## 7. 对后续改造的意义

后续引入本地缓存时，需要对比的不是“有无缓存”，而是以下两个版本：

- Baseline：Redis 单层缓存
- After：L1 本地缓存 + L2 Redis 二级缓存

后续重点观察指标：

- Avg
- TP95
- TP99
- 错误率
- 吞吐量
- Redis OPS

## 8. 后续建议

下一步建议按以下顺序推进：

1. 开始引入本地缓存
2. 优先改造：
   - `GET /shop/{id}`
   - `GET /shop-type/list`
3. 完成改造后，在同样条件下进行复测
4. 对比 Redis 单层缓存与二级缓存的差异

## 9. 当前结论摘要

本次 `S1` 基线压测已经可以确认：

- 当前 `shop` 查询已经有 Redis 缓存
- 当前还没有本地缓存
- 当前版本在有效测试下稳定
- 可以进入下一阶段，开始尝试引入本地缓存
