# hm-dianping 二级缓存改造后验证报告

## 1. 报告目的

本报告用于记录 `hm-dianping` 项目在引入本地缓存后的验证结果，确认第一版二级缓存是否已经真正生效，并为后续压测对比提供依据。

本次验证重点关注两个热点读接口：

- `GET /shop/{id}`
- `GET /shop-type/list`

## 2. 改造背景

在本次改造前，项目已经具备 Redis 单层缓存能力：

- `shop/{id}` 已通过 `CacheClient` 走 Redis 缓存
- `shop-type/list` 已通过手写逻辑走 Redis 缓存

本次改造目标不是“从无缓存到有缓存”，而是：

**在原有 Redis 单层缓存前增加一层本地缓存，形成 L1 本地缓存 + L2 Redis 二级缓存结构。**

## 3. 本次改造内容

本次第一版改造已完成以下内容：

- 引入 `Caffeine` 作为本地缓存实现
- 为 `shop/{id}` 接入本地缓存
- 为 `shop-type/list` 接入本地缓存
- 保留 Redis 作为二级缓存
- 在 `shop` 更新时，增加本地缓存失效

当前缓存链路如下：

### 3.1 `GET /shop/{id}`

1. 查 L1 本地缓存
2. L1 未命中则查 Redis
3. Redis 未命中则查数据库
4. 数据回填 Redis 与 L1

### 3.2 `GET /shop-type/list`

1. 查 L1 本地缓存
2. L1 未命中则查 Redis
3. Redis 未命中则查数据库
4. 数据回填 Redis 与 L1

## 4. 验证方式

本轮验证采用以下方式进行：

1. 对 `shop/{id}` 与 `shop-type/list` 发起压测请求
2. 在缓存工具层临时增加缓存命中日志
3. 观察应用日志，确认请求是否命中本地缓存
4. 结合压测结果，检查错误率与稳定性

说明：

- 日志验证完成后，临时日志已移除，避免影响后续正式压测

## 5. 验证结果

### 5.1 `GET /shop/{id}` 验证结果

验证现象：

- 应用日志中出现大量：

`L1 cache hit, key=cache:shop:1`

说明：

- `shop/{id}` 请求已优先命中本地缓存
- 本地缓存链路已经实际生效

压测结果特征：

- 接口响应稳定
- 错误率维持为 `0%`
- 在当前压测配置下可正常运行

### 5.2 `GET /shop-type/list` 验证结果

验证现象：

- 应用日志中出现 `shop-type:list` 对应的本地缓存命中

说明：

- `shop-type/list` 请求也已命中本地缓存
- 第二条核心热点读链路也已跑通

压测结果：

| 场景 | 接口 | 样本数 | 平均响应(ms) | 最大响应(ms) | 错误率 | 吞吐量 |
|---|---|---:|---:|---:|---:|---:|
| S2 After | `GET /shop-type/list` | 4000 | 0 | 154 | 0.00% | 209.4/sec |

## 6. 结论

本轮验证可以明确得出以下结论：

1. 第一版二级缓存已经成功接入项目
2. `shop/{id}` 已命中 L1 本地缓存
3. `shop-type/list` 已命中 L1 本地缓存
4. 两条核心热点读接口的二级缓存链路均已生效
5. 当前验证过程中未出现明显错误，接口稳定性正常

## 7. 当前阶段的意义

本次验证的意义在于：

- 证明本地缓存不是“只写了代码”，而是已经真正被命中
- 证明二级缓存链路已经在运行
- 为后续正式对比压测提供了“改造后版本”依据

换句话说，当前我们已经完成了：

- 基线确认
- 二级缓存方案落地
- 改造后功能验证

## 9. 总结

本次第一版二级缓存改造已经通过日志与压测验证，说明：

- `shop/{id}` 的 L1 本地缓存已生效
- `shop-type/list` 的 L1 本地缓存已生效
- 当前项目已经从 Redis 单层缓存升级为可工作的二级缓存结构

这意味着我们已经可以进入下一阶段：
