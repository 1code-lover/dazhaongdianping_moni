# 黑马点评项目技术问答

## 1. 项目简单介绍

**黑马点评**是一个类似小红书的本地生活点评平台，主要功能包括：
- **用户模块**：登录注册、验证码、签到、用户信息
- **探店笔记**：点赞、评论、关注、推流
- **商户模块**：商户查询、商户类型
- **优惠券**：普通优惠卷、秒杀卷
- **Redis缓存**：本地缓存(Caffeine) + Redis + 缓存击穿/雪崩/失效策略
- **消息队列**：Redis Stream → Kafka 异步下单
- **秒杀系统**：Lua脚本限流、库存预热

## 2. 点赞ZSet是什么？

### 2.1 实现原理

在 `BlogServiceImpl.java:98-122` 中，使用 **ZSet(有序集合)** 来实现点赞功能：

```java
// key = "blog:liked:" + blogId
// value = 用户ID
// score = 点赞时间戳（用于排序）
stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
```

### 2.2 为什么用ZSet而不是Set？

1. **去重**：同一用户不能重复点赞（ZSet天然去重）
2. **排序**：可用于展示"最近点赞的用户"
3. **高效查询**：`score()` 可O(1)判断某用户是否点赞
4. **获取Top**：`zrange()` 快速获取点赞列表

## 3. 排行榜大-key改进方案

### 3.1 当前问题

单个 `blog:liked:{id}` key 存储所有点赞用户，量级大时：
- 内存压力：百万粉丝的博客会有几百万条记录
- 热度不均：头部博客点赞量巨大
- 操作阻塞：ZSet操作是O(logN)

### 3.2 改进方案

#### 方案一：Redis Key拆分（分片）

**按时间分片**：
```
blog:liked:{blogId}:2024-01  → 1月点赞用户
blog:liked:{blogId}:2024-02  → 2月点赞用户
```

**按用户ID范围分片**：
```
blog:liked:{blogId}:0     → userId % 10 == 0
blog:liked:{blogId}:1     → userId % 10 == 1
...
```

优点：降低单key数据量，支持归档清理

#### 方案二：使用Redis Hash

点赞状态只存储"是否点赞"，不关心时间：
```
blog:liked:{blogId} → {userId: 1}  # Hash存储
```
O(1) 判断，O(1) 写入，但无法获取"点赞列表"

#### 方案三：Redis Cluster方案

- **垂直拆分**：点赞状态用Redis Hash，点赞列表用专门的用户服务
- **冷热分离**：最近1年的点赞用Redis，历史归档到数据库
- **布隆过滤器**：快速判断是否点赞过，减少Redis请求

### 3.3 多节点方案

项目是**无状态服务**，可水平扩展，但本地缓存有状态：

| 方案 | 说明 |
|-----|-----|
| 分布式缓存 | 用Redis统一缓存，本地缓存做一级（推荐） |
| 缓存广播 | 缓存更新时发布到所有节点 |
| 哈希一致性 | 同一blog走同一节点（不推荐，破坏无状态） |

## 5. 为什么引入Kafka？

### 5.1 当前架构

```
用户下单 → Redis Stream → Relay → Kafka → 消费者 → 数据库
```

### 5.2 为什么需要Kafka？

项目已经有 `redis_stream`，为什么还要Kafka？

| 维度 | Redis Stream | Kafka |
|-----|-------------|------|
| **持久化** | 有限（内存） | 无限（磁��） |
| **吞吐量** | 一般 | 极高 |
| **分区/顺序** | 不支持 | 支持 |
| **消费者组** | 一般 | 成熟 |
| **生态集成** | 无 | 丰富（对接Flink等） |
| **消息回溯** | 弱 | 强 |

### 5.3 业务场景

项目用Kafka处理**秒杀订单**：
- 瞬间高并发：10万/秒
- 需要削峰填谷：异步处理
- 需要持久化：防止订单丢失
- 需要重试/死信：订单失败处理

## 6. Kafka基本概念

### 6.1 核心角色

```
Broker          → Kafka服务实例，一个Broker一个topic的多个分区
Topic          → 消息主题（订单、日志）
Partition     → 分区，物理存储，一个分区一个消费者组
Replica       → 副本，高可用
Producer      → 生产者
Consumer      → 消费者
Consumer Group → 消费者组，同一组内每个消息只消费一次
```

### 6.2 消息语义

| 语义 | 说明 | 项目中配置 |
|-----|-----|----------|
| **at-least-once** | 至少一次（可能重复） | 默认 |
| **at-most-once** | 至多一次（可能丢失） | 较少用 |
| **exactly-once** | 精确一次 | 需事务 |
| **幂等性** | 开启 `enable.idempotence=true` | 当前用SeekToCurrent |

### 6.3 项目配置

`KafkaConfig.java:48-61`：
```java
// 手动ack：收到消息后立即ack
factory.getContainerProperties().setAckMode(AckMode.MANUAL_IMMEDIATE);

// 失败重试：3次，间隔1秒
factory.setErrorHandler(new SeekToCurrentErrorHandler(recoverer, new FixedBackOff(1000L, 3L)));

// 死信：发送到 topic-dlt
DeadLetterPublishingRecoverer → 原topic-dlt分区
```

### 6.4 重复发送问题

当前方案：
1. **Producer端**：Kafka幂等性（项目未开启）
2. **Consumer端**：去重表（订单ID唯一）、SeekToCurrent重试
3. **最佳实践**：结合数据库事务 + 幂等性

## 7. 本地缓存设计

### 7.1 存储什么数据？

`LocalCacheConfig.java`：

| 缓存 | 存储内容 | 大小 | TTL |
|-----|--------|-----|-----|
| shopLocalCache | 商户信息 | 1000条 | 5分钟 |
| shopTypeLocalCache | 商户类型列表 | 100条 | 10分钟 |

**为什么存这些？**
- 访问频繁：热点商户数据
- 查询量大：首页、商户详情页
- 变化少：商户信息不常变

### 7.2 更新机制

`ShopServiceImpl.java`：
```java
// 1. 先更新数据库
// 2. 删除Redis缓存
// 3. 发布缓存失效消息
cacheInvalidationPublisher.publishShopInvalidation(shopId);
```

`CacheInvalidationListener.java:28-47`：
```java
// 监听Redis Pub/Sub，删除本地缓存
onMessage() → cacheClient.deleteShopLocalCache(shopId);
```

### 7.3 多节点部署问题

项目是无状态服务，可部署多个实例。

**问题**：每个实例有独立的本地缓存，缓存不一致

**解决方案**：项目使用 **Redis Pub/Sub 广播机制**

```
实例A更新数据 → Redis发布消息 "cache:invalidate" → 实例A/B/C都删除本地缓存
```

### 7.4 多节点本地缓存处理方案

| 方案 | 说明 | 适用场景 |
|-----|------|---------|
| **广播失效（推荐）** | Redis Pub/Sub通知所有节点删除缓存 | 项目当前方案 |
| **缓存不设本地** | 仅用Redis，降低一致性要求 | 数据一致性要求高 |
| **一致性哈希** | 同一数据走同一节点 | 不适合无状态服务 |
| **共享存储** | 用Redis替代本地缓存 | 推荐 |

## 10. 场景题：帖子点赞评论量设计

### 10.1 需求分析

"帖子展示点赞和评论量"

### 10.2 技术方案

#### 数据模型

```java
Blog {
    Long id;
    Long userId;
    String title;
    String content;
    Integer liked;     // 点赞数
    Integer comment; // 评论数
}
```

#### 实现方案一：数据库存储（当前）

```java
// 点赞时
update blog set liked = liked + 1 where id = ?
// Redis ZSet记录点赞用户（用于去重+列表）
```

**优点**：简单可靠
**缺点**：热点博客QPS高时DB压力大

#### 实现方案二：Redis计数（高性能）

```java
// key: blog:stat:{blogId}
// hash: {liked: 100, comment: 50}
stringRedisTemplate.opsForHash().increment("blog:stat:" + blogId, "liked", 1);
```

**优点**：高性能
**缺点**：数据可能丢失，需定期回写DB

#### 实现方案三：混合方案（推荐）

| 数据 | 存储 | 说明 |
|-----|------|-----|
| 点赞数 | Redis计数器 | 高性能读取 |
| 评论数 | 数据库 | 需要列表功能 |
| 点赞列表 | ZSet | 展示"谁点赞了" |
| 评论列表 | 数据库分页 | 评论可多可回复 |

```java
// 读取时
int liked = (int)stringRedisTemplate.opsForHash().get("blog:stat:" + blogId, "liked");
// 更新时（异步或同步）
```

#### 性能优化

| 优化点 | 方案 |
|-----|-----|
| 点赞QPS高 | Redis计数器 + 异步落库 |
| 评论列表大 | 分页查询 + 懒加载 |
| 热点博客 | 本地缓存(Caffeine) |
| 防止刷赞 | ZSet去重 + 限流 |

### 10.3 完整设计

```java
// BlogServiceImpl - 点赞
public Result likeBlog(Long blogId) {
    String likedKey = "blog:liked:" + blogId;
    String statKey = "blog:stat:" + blogId;

    // 1. 检查是否点赞（ZSet去重）
    if (stringRedisTemplate.opsForZSet().score(likedKey, userId) != null) {
        return Result.fail("已点赞");
    }

    // 2. 写入点赞列表
    stringRedisTemplate.opsForZSet().add(likedKey, userId, System.currentTimeMillis());

    // 3. 点赞数+1（Redis计数）
    stringRedisTemplate.opsForHash().increment(statKey, "liked", 1);

    // 4. 异步落库（可选）
    asyncService.saveLikeLog(blogId, userId);

    return Result.ok();
}

// 读取
public BlogVO getBlog(Long blogId) {
    Blog blog = blogMapper.selectById(blogId);
    long liked = (long)stringRedisTemplate.opsForHash().get("blog:stat:" + blogId, "liked");
    blog.setLiked((int)liked);
    return blog;
}
```
